入口与准备阶段

一条用户消息进来,在进入 while 循环之前,Hermes 先做了一大堆「每轮一次」的准备工作。这一篇追踪从 chat() 到循环开始之间发生的一切。

两个入口:都是薄转发

对外暴露的入口在 run_agent.py 的 AIAgent 类上,但它们几乎不做事,只是转发到 agent/conversation_loop.py 的真身:

# run_agent.py
def chat(self, message, stream_callback=None) -> str:
    result = self.run_conversation(message, stream_callback=stream_callback)
    return result["final_response"]          # 只取最终文本

def run_conversation(self, user_message, system_message=None,
                     conversation_history=None, task_id=None, ...) -> dict:
    from agent.conversation_loop import run_conversation
    return run_conversation(self, user_message, ...)   # 转发到真身
run_agent.py · AIAgent.chat() / run_conversation()

区别:chat() 是「省心接口」,只返回最终回答字符串;run_conversation() 返回完整结果字典(含 final_response、messages、api_calls 等)。前端要展示中间过程就用后者。

这正是「窄入口」设计:无论 CLI、网关还是 TUI,都调这同一个 run_conversation(),所以核心逻辑只有一份。

同步返回 vs 流式回吐(易混淆点)

run_conversation() 是同步阻塞的:它把整轮多次工具调用全部跑完才返回。也就是说,final_response 是「所有工具都执行完、模型给出最终答复之后」的结果——中途的工具结果不会作为返回值吐出,只是被 append 进 messages 喂回模型。

但「返回值要等到最后」并不代表「用户也要等到最后才看到字」。靠 stream_callback,模型生成最终答复时会逐块(delta)回吐文本:

result = self.run_conversation(message, stream_callback=on_delta)
#                                        ↑ 每来一小段文字就触发,边生成边显示
return result["final_response"]   # 这一行要等整轮结束才执行
run_agent.py · chat(stream_callback=...)
维度行为
函数返回值整轮跑完才返回(同步阻塞)
stream_callback生成最终答复时逐块回吐,前端可「实时打字」
中途工具结果不作为返回值,只 append 进 messages 喂回模型
两种「提前」结束也会给出 final_response:被用户中断(/stop)返回当时的部分结果;迭代预算耗尽时由 _handle_max_iterations 生成一段总结。详见第 2、5 篇。

准备阶段 = build_turn_context

进入 conversation_loop.run_conversation() 后,第一件大事是把所有「每轮一次」的准备工作交给一个函数:

# agent/conversation_loop.py
_ctx = build_turn_context(agent, user_message, system_message,
                          conversation_history, task_id, ...)
user_message       = _ctx.user_message
messages           = _ctx.messages
active_system_prompt = _ctx.active_system_prompt
effective_task_id  = _ctx.effective_task_id
turn_id            = _ctx.turn_id
_should_review_memory = _ctx.should_review_memory
_ext_prefetch_cache   = _ctx.ext_prefetch_cache
# … 循环要用的局部变量都从 _ctx 里取回
agent/conversation_loop.py → agent/turn_context.py · build_turn_context()

为什么要抽出来?因为这段准备逻辑有上百行,把它从循环函数里挪走,主循环才看得清。build_turn_context 会原地修改 agent 对象(设置一堆 agent._xxx 状态),并返回一个 TurnContext 给循环读回。

准备阶段都做了什么(按真实顺序)

build_turn_context 里有近百行,但本质就是把「真正开始对话前」该铺垫的事一次做完。按源码顺序,可以归成 6 个阶段。下面每条都说清「做什么 + 为什么」。

阶段 1 · 环境与线程准备

让这一轮跑在正确的运行环境里,并把日志/副任务接上当前会话。

  • 安全 stdio —— install_safe_stdio()。给标准输出/错误套一层保护,防止在无终端的 daemon / headless 环境里因「管道断了」(broken pipe)抛 OSError 把整个进程搞崩。
  • 告诉副 LLM 当前主模型 —— set_runtime_main(provider, model, ...)。Hermes 有些「side 任务」(生成标题、向量化、视觉等)会另起一个小模型;这里把本轮的主提供商/模型告诉它们,免得用错后端。
  • 给日志贴会话标签 —— set_session_context(session_id)。本线程之后打的所有日志都带上这个 session ID,hermes logs --session 才能按会话筛。
  • 恢复主提供商 —— _restore_primary_runtime()。如果上一轮因为主提供商故障切到了备用,这里先切回主,保证新一轮从首选模型开始。

阶段 2 · 纳入新工具(缓存安全)

  • 轮次间刷新 MCP 工具 —— refresh_agent_mcp_tools()。有些 MCP 服务器冷启动要 2–6 秒,上一轮可能还没连上;在本轮组装 tools= 之前刷新一次,把新连上的工具纳进来。之所以缓存安全,是因为它只在「每轮开头」运行,只会延长一个全新请求的前缀,绝不改动进行中那一轮已缓存的前缀。没有 MCP 服务器时直接跳过。

阶段 3 · 清洗输入 + 生成本轮标识

  • 剥代理字符(surrogate) —— 某些模型(经 Ollama 的 Kimi、GLM、Qwen)会吐出孤立的 surrogate 字符,后面 json.dumps() 一碰就崩。这里提前把用户输入里的剥掉,省去之后的 3 次重试。
  • 存流式回调与持久化覆盖项 —— 把 stream_callback、persist_user_message 等挂到 agent 上,供后面的 API 调用和落库取用。
  • 生成 task_id / turn_id —— effective_task_id = task_id or uuid4(),turn_id = "session:task:hex"。task_id 用来隔离并发任务各自的终端/浏览器会话;turn_id 唯一标识这一轮,贯穿日志与请求。
三个 ID 的区别(由大到小):
ID粒度作用
session_id一整段会话(跨多轮)缓存/记忆/历史的锚点;hermes logs --session 按它筛
task_id一个任务(一次 run_conversation)隔离资源:并发任务各自的终端/浏览器互不串
turn_id一轮(一问一答)= session:task:hex,把本轮所有日志串起来便于排查
api_request_id一轮内一次调模型= turn_id:api:N(见第 3 篇)
打比方:session 是一本书,turn 是一章,api_request 是章里的一段;而 task_id 不是层级,是给这一章配的「独立工作台」,保证并发任务不会把工具/终端搞混。
什么是 surrogate(代理)字符? Unicode 里超出基本平面的字符(如 emoji 😀、生僻字)在 UTF-16 编码下要用两个码位「成对」表示,这一对就叫 surrogate pair(高位 U+D800–U+DBFF + 低位 U+DC00–U+DFFF)。这两个码位本身不是合法字符,必须成对出现才有意义。当模型输出时只吐出了「半个」——一个落单的高位或低位 surrogate(lone surrogate),它就是个非法字符:Python 的 json.dumps() 在序列化时会直接抛 UnicodeEncodeError 崩掉。所以这里先把这些「半个字符」剥掉,避免后续 API 调用因为一个残缺字符而反复失败。

阶段 4 · 重置本轮状态

把上一轮残留的计数和连接清干净,给这一轮一个干净起点。

  • 归零所有重试计数 —— _invalid_tool_retries、_invalid_json_retries、_empty_content_retries…… 全部清零,并重置工具守卫和 vision 支持标志。第 3 篇讲重试时这些计数才有意义。
  • 清理死连接 —— _cleanup_dead_connections()。上一轮 provider 出问题可能留下半死的 TCP 连接,这里清掉,用全新连接开始。
  • 新建迭代预算 —— agent.iteration_budget = IterationBudget(max_iterations)。每轮一个全新预算对象,就是主循环的安全阀(第 2 篇详讲)。

阶段 5 · 从历史恢复状态与计数

  • 复制历史 —— messages = list(conversation_history)。拷一份再用,绝不直接改调用方传进来的列表。
    为什么?这个 conversation_history 是调用方(CLI / 网关 / TUI)持有并会复用的对象。而本轮处理过程中,messages 会被不断追加内部消息——user 消息、每个工具的调用与结果、assistant 的中间回复……如果直接在原列表上改,这些「内部草稿」就会泄漏回调用方:轻则它的界面/历史显示多出一堆中间消息,重则同一条 user 消息被重复追加、或并发/重试时两边互相污染。拷贝一份是「不修改入参」的基本约定——本轮怎么折腾都只动副本,调用方的原始历史保持干净,由收尾阶段统一、可控地落库。
  • 恢复 todo —— 从历史里把之前的 todo 列表重新加载进 todo store(断点续聊也能接上)。
  • 恢复提醒周期 —— 数一数历史里有多少条 user 消息,据此还原「该提醒复盘记忆了吗」的周期计数;再累加判断本轮要不要复盘,得出 should_review_memory(循环结束后用)。
  • 追加用户消息 —— 把 {"role":"user", ...} append 进 messages 并记下标;同时保留一份「干净原文」original_user_message,供转录和记忆查询。
    为什么要留干净原文?真正发给模型时,这条 user 消息会被临时拼上一些参考资料(检索回来的相关记忆、插件加的上下文等,详见下方机制 A);另外它也可能带一些 API 专用的合成前缀。这些都是「给模型看的」,不该混进你的对话记录。所以系统另存一份你真正说的话,记忆/历史/转录都用它,保持干净。
两套别混淆的机制:消息注入 vs 后台复盘
机制 A · 注入参考资料机制 B · 记忆复盘 nudge
做什么把检索回来的相关记忆 + 插件上下文,临时拼到 user 消息后面周期性提醒「该把学到的写进记忆了」
谁处理当前正在对话的模型当场读、当场用(当作参考)另起一个后台 agent(单独的模型调用)去复盘
何时本轮调模型时(每圈都拼,只改副本)本轮答复发出之后,后台跑,不打断对话
代码循环里 _ext_prefetch_cache + _plugin_user_context 注入should_review_memory → _spawn_background_review
nudge(轻推)本意是「温和提醒」,在 Hermes 里主要就是机制 B 那种周期性提醒(复盘记忆 / 更新技能),由后台 agent 处理——不是塞进当前消息让你正在对话的模型处理。容易误解的点在这:被拼进消息的是「参考资料」(机制 A),不是「提醒」(机制 B)。
「提醒复盘记忆」是什么周期? Hermes 希望 agent 每隔 N 轮(_memory_nudge_interval)停下来把学到的东西写进长期记忆,但不能每轮都提醒。所以用 _turns_since_memory 数「距上次复盘过了几轮」,到 N 就置 should_review_memory=True 并清零——这个标志在收尾阶段(第 5 篇)触发后台另一个 agent 去复盘(即上方机制 B),不打断当前回答。

为什么要「从历史还原」? 断点续聊时内存计数器是全新的 0,但会话其实已经聊了很多轮,直接从 0 数会让提醒时机错位。于是用 历史 user 轮数 % N 把计数器拨回正确位置。例:周期 5、已聊 12 轮 → 12 % 5 = 2 → 计数器设为 2,再过 3 轮提醒,节奏接着之前走,而不是从头重数。

阶段 6 · 系统提示 + 落库 + 压缩

这是准备阶段的收官:把「身份说明」备好、把这轮输入存进数据库、必要时给历史「瘦身」,让循环可以安全地开始第一次调模型。

  • 恢复或构建 System Prompt
    system prompt 是发给模型的「开场说明书」——告诉它你是谁、有哪些工具、规则是什么,放在每次请求的最前面。这里的逻辑是 if agent._cached_system_prompt is None: restore_or_build_system_prompt(...):只有当缓存为空(本会话第一轮)才真正构建它;一旦建好就缓存在 agent._cached_system_prompt 并存进数据库,之后每一轮都原样复用同一份。
    为什么重要:system prompt 在请求最前面,是缓存前缀的第一段。只要它逐字节不变,后面所有轮都能命中缓存。这就是「prompt 缓存铁律」的实现点——绝不中途重建。
  • 现在才建「DB 会话行」
    Hermes 用 SQLite 数据库(SessionDB,文件在 ~/.hermes/ 下)持久化对话:一行(row)代表一段会话,行里的各列存着 session_id、system_prompt、消息历史、模型等。_ensure_db_session() 就是「确保这一行存在」(没有就插入,有就跳过)。
    为什么推迟到这一步:建行时会把 system prompt 一起写进那一列。如果在 system prompt 还没准备好时就建行,这列只能写成 NULL(空);下次加载会话时代码发现「存的 prompt 是空的」,就重建它并打警告,还会让首轮缓存白白未命中(Issue #45499)。所以特意安排成「先备好 prompt,再建行」。
  • 崩溃韧性「落库」
    「落库」就是把数据写进数据库。_persist_session(...) 在会话行一就绪时,马上把这条 user 消息写进 SQLite。
    为什么现在就写:万一接下来调模型时进程崩了 / 断电,这轮你刚输入的内容也已经存下来了,重启后能恢复,不会丢。
  • 预检上下文压缩
    「上下文压缩」= 当历史太长、快超过模型能接收的上限时,把旧消息总结/精简掉,腾出空间。这一步先用一个便宜的预检判断「这轮可能超限吗」,只有可能超限时才去做昂贵的完整 token 估算和真正的压缩。
    注意:这是整个流程里唯一允许改动历史上下文的地方(其余环节都不能动历史,否则破坏缓存)。压缩有可能新建一个会话、把 conversation_history 重置为 None。
  • pre_llm_call 钩子 + 外部记忆预取
    准备阶段的最后,留了两个「插进自定义逻辑」的扩展点:
    • pre_llm_call 钩子 —— 「钩子(hook)」是框架预留的一个挂载点:到了某个时机,框架会回头调用你注册的函数,让你插一脚而不必改核心代码。pre_llm_call 就是「**每次准备调模型之前**」这个时机。插件可以在这里返回一段额外上下文,框架会把它拼到 user 消息后面一起发给模型。
      举例:一个「当前时间/天气」插件,可以在每轮调模型前自动塞一句「现在是 2026-07-01 北京晴」,让模型始终知道实时信息。
    • 外部记忆预取 —— Hermes 的长期记忆可以接外部记忆后端(可插拔的第三方服务,如 mem0、honcho),它们把跨会话的记忆存在自己的库里,用时按当前问题检索相关片段。「预取(prefetch)」= 提前查一次:在进循环前,就用本轮的问题去外部后端捞回相关记忆,存进 ext_prefetch_cache。
      为什么要缓存起来:记得吗——一轮里要调好多次模型。如果每圈都去外部后端查一遍,既慢又浪费。所以「一轮查一次、整轮复用」:预取的结果缓存在 ext_prefetch_cache 里,循环里每次调用直接拿现成的。

交给循环的「上下文对象」

准备完,build_turn_context 返回一个 TurnContext。循环只从它里面取这些值:

字段含义
user_message清洗后的用户消息(可能带 API 专用合成前缀;参考资料在调模型时才临时拼上)
original_user_message干净原文,供转录/记忆查询
messages本轮的消息列表(已 append 用户消息)
active_system_prompt本轮生效的 system prompt(缓存的)
effective_task_id / turn_id任务/轮次唯一标识
should_review_memory本轮结束后要不要提醒复盘记忆
ext_prefetch_cache外部记忆预取结果,循环内复用

循环开始前的纯局部变量

拿回上下文后,run_conversation 再初始化几个纯局部计数器(只在循环里用,不挂到 agent 上):

api_call_count = 0          # 已发起的 API 调用数
final_response = None        # 最终答复(循环里填)
interrupted = False          # 是否被用户中断
failed = False               # 是否失败退出
compression_attempts = 0     # 循环内压缩尝试次数
_turn_exit_reason = "unknown"  # 诊断:循环为何结束
agent._auth_pool_refresh_counts = {}  # 凭证刷新计数,防止 401 空转
agent/conversation_loop.py · run_conversation()

本篇要点

  • 入口 chat() / run_conversation() 只是薄转发,真身在 conversation_loop.py。
  • 所有「每轮一次」的准备都集中在 build_turn_context,它改 agent 状态并返回 TurnContext。
  • System prompt 只构建一次、全程缓存——缓存铁律的实现点。
  • 用户消息一进来就落库,保证崩溃可恢复。
  • 压缩是唯一允许改历史的环节,且在进入循环前完成。
