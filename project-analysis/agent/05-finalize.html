收尾阶段

循环跳出后(拿到答复、被中断、或预算耗尽),还要做一连串收尾:落库、转换输出、组装结果字典、安排后台复盘。这些全集中在 finalize_turn。

收尾的入口

循环结束后,run_conversation 把一切交给一个函数,它的返回值就是整个 run_conversation 的返回值:

# agent/conversation_loop.py(循环之后)
result = finalize_turn(agent, final_response, messages, ...,
                       _should_review_memory=..., interrupted=..., ...)
return result
agent/conversation_loop.py → agent/turn_finalizer.py · finalize_turn()
finalize_turn 是同步、单一返回的(没有 await、没有提前 return),刻意镜像它替换掉的那段内联代码。

收尾都做了什么(按真实顺序)

  1. 没答复?先处理「迭代耗尽」

    if final_response is None and 预算/迭代耗尽: _handle_max_iterations(...)

    如果循环跑完了却没拿到 final_response(被工具循环占满),调 _handle_max_iterations 生成一段总结作为答复,而不是返回空。

  2. 容错收尾区(关键设计)

    轨迹保存、资源拆除、会话落库都是可能抛异常的操作(SQLite 写入等)。它们被包在一个区域里,任何一个抛错都收集进 _cleanup_errors 而非崩溃——保证用户等待的 final_response 一定能返回(#8049)。

  3. 收尾被中断的工具序列

    close_interrupted_tool_sequence(messages, final_response)

    若中途被中断,可能留下「有 tool_call 但没 tool 结果」的悬空结构。这里补齐,避免落库出不合法的消息序列。

  4. 持久化会话

    agent._persist_session(messages, conversation_history)

    把完整一轮写入 JSON 日志 + SQLite。这是私有重试之后的最终落库。

  5. 追加运行时页脚

    若有答复且未被中断,可能在末尾追加一段运行时页脚(footer)。

  6. transform_llm_output 插件钩子

    hermes_cli.plugins · transform_llm_output

    插件可在返回前改写答复文本。第一个返回非空字符串的钩子生效,返回 None/空则保持原文。

  7. post_llm_call 插件钩子

    hermes_cli.plugins · post_llm_call

    本轮工具循环结束后触发一次,插件常用它把对话数据同步到外部记忆系统。

  8. 提取本轮 reasoning

    从后往前找最近一条非空 reasoning,但遇到本轮起始的 user 消息就停——绝不跨进上一轮,避免 UI 推理框显示陈旧内容(#17055)。

  9. 组装结果字典

    把所有东西打包成 result(下节详列)。

  10. 清中断状态、清流回调

    agent.clear_interrupt();agent._stream_callback = None

    清掉中断标志和流式回调,防止泄漏到下一次调用。

  11. 判断是否该复盘技能

    _should_review_skills(基于本轮 _iters_since_skill)

    本轮用了多少工具迭代,决定是否到了提醒复盘技能的周期。

  12. 同步外部记忆 + 排下次预取

    agent._sync_external_memory_for_turn(...)

    把完成的这一轮同步给外部记忆提供商,并排队下一次预取。

  13. 安排后台复盘(不阻塞返回)

    agent._spawn_background_review(...)

    若需复盘记忆或技能,在答复交付之后才后台跑——绝不与用户任务争抢模型注意力。尽力而为,失败不影响主流程。

返回的结果字典

finalize_turn 返回的 result 远不止 final_response,还带了一整轮的元数据与计量:

{
  "final_response": ...,        # 最终答复文本
  "last_reasoning": ...,        # 本轮推理(供 UI 推理框)
  "messages": [...],            # 完整消息历史
  "api_calls": N,               # 本轮发起的 API 调用数
  "completed": bool,            # 是否正常完成
  "interrupted": bool,          # 是否被中断
  "failed": bool,
  "turn_exit_reason": "...",    # 循环为何结束(见第 2 篇)
  "model": ..., "provider": ..., "base_url": ...,
  # ── 计量(整段会话累计)──
  "input_tokens": ..., "output_tokens": ...,
  "cache_read_tokens": ..., "cache_write_tokens": ...,
  "reasoning_tokens": ..., "total_tokens": ...,
  "estimated_cost_usd": ..., "cost_status": ..., "cost_source": ...,
  "session_id": ...,
}
agent/turn_finalizer.py · result 字典

另外几个条件字段:守卫拦停时带 guardrail;收尾有异常时带 cleanup_errors;若答复后才到的 /steer 没处可塞,带 pending_steer 让调用方作为下一轮交付。

注意:这里不结束会话

收尾不调用记忆提供商的 on_session_end() / shutdown_all()。因为多轮会话里 run_conversation() 每条用户消息都会被调一次,每轮都关闭会杀死提供商,导致第二条消息没法用。真正的会话结束清理交给 CLI(atexit / /reset)和网关(会话过期 / _reset_session)。

本篇要点

  • 收尾全在 finalize_turn:落库、输出转换、组装结果、安排复盘。
  • 收尾区容错:任何清理异常都收集进 cleanup_errors,绝不吞掉用户的答复。
  • 返回的是富结果字典:答复 + 推理 + 消息 + token 计量 + 成本 + 退出原因。
  • 后台记忆/技能复盘在答复之后才跑,不抢模型。
  • 这里不结束会话,多轮对话靠 CLI / 网关在真正结束时清理。
