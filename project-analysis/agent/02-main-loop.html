主循环机制

准备就绪后进入 while 循环。这一篇只讲循环本身的「骨架」:它靠什么条件转、靠什么安全退出,以及每一圈开头做的固定动作。具体的「调模型」和「跑工具」留给第 3、4 篇。

循环条件

while (api_call_count < agent.max_iterations
       and agent.iteration_budget.remaining > 0) \
      or agent._budget_grace_call:
    ...
agent/conversation_loop.py · run_conversation()

三个闸门,任意一个挡住就停(除非有 grace):

  • api_call_count < max_iterations —— 调用次数没到上限(默认 90)。
  • iteration_budget.remaining > 0 —— 迭代预算还有余。
  • or agent._budget_grace_call —— 即使前两个不满足,只要挂着「宽限」标志,就再放一圈。
为什么有两个看起来重复的上限(max_iterations 和 iteration_budget)?因为预算是跨父子 agent 协调的:父用 max_iterations(默认 90),每个子 agent 有独立预算(默认 50)。而 execute_code 这类编程式工具调用会退还预算,不占额度——这就需要一个能 consume/refund 的独立计数器,而不只是一个简单的 api_call_count。

每一圈开头的固定动作

进入循环体后,在真正调模型之前,每一圈都先做这几件事:

  1. 开启新一圈的检查点去重

    agent._checkpoint_mgr.new_turn()

    重置本轮检查点去重,让这一圈最多只拍一次快照。

  2. 检查中断

    if agent._interrupt_requested: break

    用户发了新消息 / 按了 /stop,这里立刻跳出循环,把 interrupted=True、_turn_exit_reason="interrupted_by_user"。中断检查放在每圈最开头,保证响应及时。

  3. 计数 +1,记活动

    api_call_count += 1;agent._touch_activity("starting API call #N")

    _touch_activity 更新「最后活动时间」,供网关的不活动超时监控用。

  4. 消费预算 / 消费 grace 标志

    若挂着 _budget_grace_call,清掉它(这一圈跑完无论如何都退);否则调 iteration_budget.consume()——返回 False 说明预算用尽,置 _turn_exit_reason="budget_exhausted" 并 break。

  5. 触发 step_callback(网关钩子)

    agent.step_callback(api_call_count, prev_tools)

    若设了回调,发出 agent:step 事件(网关用它做进度推送),并附上上一圈的工具调用摘要。

迭代预算:循环的安全阀

IterationBudget 本身很简单,就是一个线程安全的计数器:

# agent/iteration_budget.py
class IterationBudget:
    def __init__(self, max_total):
        self.max_total = max_total
        self._used = 0
        self._lock = threading.Lock()

    def consume(self) -> bool:        # 试消费一次,超额返回 False
        with self._lock:
            if self._used >= self.max_total:
                return False
            self._used += 1
            return True

    def refund(self):                 # 退还一次(execute_code 用)
        with self._lock:
            if self._used > 0:
                self._used -= 1

    @property
    def remaining(self):
        return max(0, self.max_total - self._used)

因为每个子 agent 有独立预算,所以「父 + 所有子」的总迭代数可以超过父的 90——这是有意的设计,让委派出去的子任务有自己的预算空间。

refund 的用处:execute_code(在沙箱里编程式地调工具)的那一圈会被退还,不计入预算。否则一次 execute_code 里跑了 20 个工具,会把额度瞬间烧光。

grace 调用:多给一次机会

预算用尽时,Hermes 不是粗暴地直接停,而是可以挂一个 _budget_grace_call 标志,再放模型一圈——让它有机会基于已有信息收个尾、给出答复,而不是把对话硬切在半截工具调用上。这一圈跑完,标志被清,循环必然退出。

循环为何退出:_turn_exit_reason

循环用一个诊断变量记录「为什么停」,排查问题时很有用:

值含义
interrupted_by_user用户中断(新消息 / /stop)
budget_exhausted迭代预算用尽
ollama_runtime_context_too_smallOllama 运行时上下文不够装工具

本篇要点

  • 循环由三个闸门守:调用次数、迭代预算、grace 标志。
  • 中断检查在每圈最开头,保证 /stop 及时生效。
  • 迭代预算是线程安全计数器,可 consume/refund;父 90、子 50,各自独立。
  • execute_code 的迭代会被退还,不占预算。
  • 预算耗尽可触发 grace 调用,给模型最后一次收尾机会。
