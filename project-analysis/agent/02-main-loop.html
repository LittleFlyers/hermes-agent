主循环机制

准备就绪后进入 while 循环。这一篇只讲循环本身的「骨架」:它靠什么条件转、靠什么安全退出,以及每一圈开头做的固定动作。具体的「调模型」和「跑工具」留给第 3、4 篇。

循环条件

while (api_call_count < agent.max_iterations
       and agent.iteration_budget.remaining > 0) \
      or agent._budget_grace_call:
    ...
agent/conversation_loop.py · run_conversation()

三个闸门,任意一个挡住就停(除非有 grace):

  • api_call_count < max_iterations —— 调用次数没到上限(默认 90)。
  • iteration_budget.remaining > 0 —— 迭代预算还有余。
  • or agent._budget_grace_call —— 即使前两个不满足,只要挂着「宽限」标志,就再放一圈。
为什么有两个看起来重复的上限(max_iterations 和 iteration_budget)?因为预算是跨父子 agent 协调的:父用 max_iterations(默认 90),每个子 agent 有独立预算(默认 50)。而 execute_code 这类编程式工具调用会退还预算,不占额度——这就需要一个能 consume/refund 的独立计数器,而不只是一个简单的 api_call_count。

每一圈开头的固定动作

进入循环体后、在真正调模型之前,每一圈都先按顺序做这 5 件事。它们都很短,但各有用处——下面逐个讲清「做什么 + 为什么」。

  1. 开启新一圈的检查点去重

    agent._checkpoint_mgr.new_turn()

    检查点(checkpoint)是 Hermes 给工作状态拍的「快照」,出问题时可以回滚到某个快照。去重(dedup)是为了防止同一圈里因为多个触发点而重复拍照。每圈开头调一次 new_turn(),等于跟检查点管理器说「新的一圈开始了」,把去重标记清零——这样这一圈最多只拍一张快照,既留住了回滚能力,又不会拍一堆冗余快照拖慢速度、占空间。

  2. 检查中断

    if agent._interrupt_requested: break

    _interrupt_requested 是一个标志位,当用户发来新消息或按了 /stop 时被别的线程置上。这里一旦发现它为真,就立刻 break 跳出整个循环,并记下 interrupted=True、_turn_exit_reason="interrupted_by_user"。
    为什么放在每圈最开头:循环里最耗时的是「调模型」和「跑工具」。把中断检查放在每圈的第一步,意味着每完成一圈、还没开始下一次昂贵操作前,都会查一次——用户按下 /stop 后最多等当前这一圈结束就能停,不会被晾很久。

  3. 计数 +1,记一次「活动」

    api_call_count += 1;agent._api_call_count = api_call_count;agent._touch_activity(...)

    api_call_count 加一(它既是循环条件的闸门,也用于日志「第 N 次调用」),并同步到 agent._api_call_count 供别处读取。_touch_activity("starting API call #N") 则更新 agent 的「最后活动时间」。
    为什么要记活动时间:网关有「不活动超时」监控——如果一个会话长时间没动静(卡死、网络挂住),它需要据此判断并清理。每圈刷新一次活动时间戳,等于告诉监控「我还活着、正在干活」,避免正常的长任务被误判为卡死。

  4. 消费预算 / 消费 grace 标志

    if grace: 清掉 grace;else if not budget.consume(): break

    这一步决定「这一圈到底准不准跑」,有两条岔路:

    • 挂着 grace 标志 —— 把 _budget_grace_call 清掉。grace 是一个「即使闸门关了也再放一圈」的预留钩子(下一节细讲;注意:核心自身不会把它置 True);用掉就清,保证这一圈跑完后循环必然退出。
    • 正常情况 —— 调 iteration_budget.consume() 真正「花掉」一次预算额度。它返回 False 说明额度已用光,于是记 _turn_exit_reason="budget_exhausted" 并 break。
    注意:第一步的 while 条件里已经粗查过 remaining > 0,这里再 consume() 是真正的「扣费」并加锁,保证父子 agent 并发时计数不出错(见下节)。

  5. 触发 step_callback(给网关推进度)

    agent.step_callback(api_call_count, prev_tools)

    如果调用方(主要是网关)注册了 step_callback,这里就发出一个 agent:step 事件,让外部能实时看到「agent 又往前走了一步」。回调带两个信息:当前是第几次调用,以及 prev_tools——上一圈调了哪些工具、各自的参数和返回结果。
    prev_tools 怎么来的:从 messages 末尾往前扫,找到最近一条带 tool_calls 的 assistant 消息,再把紧跟其后的若干 tool 结果按 tool_call_id 配对,拼成 [{name, arguments, result}, ...]。这样网关就能把「刚才跑了 read_file、结果是 XXX」实时显示给用户。整段用 try/except 包住——推送进度失败绝不能搞垮主循环。

深入:检查点(Checkpoint)机制

第 1 步的 new_turn() 背后,是一套设计得很用心的安全网。它值得单独讲透——因为 agent 会真的改你的文件,而它就是那颗「后悔药」。

tools/checkpoint_manager.py

它解决什么问题

agent 会通过 write_file、patch、带破坏性参数的 terminal 修改你的文件。万一改错、删错,你需要能一键回到之前的状态。检查点就是在每次改文件之前自动拍快照,事后可回滚。

关键定位:它不是工具,LLM 根本看不到它(源码注释:This is NOT a tool — the LLM never sees it)。它是透明的基础设施,默认关闭,靠 config.yaml 的 checkpoints 开关或 --checkpoints CLI 标志开启。

触发与去重(这就是 new_turn 的意义)

CheckpointManager 由 AIAgent 持有,核心就两个方法:

def new_turn(self):                    # 每轮开头调(主循环第 1 步)
    self._checkpointed_dirs.clear()    # 清空「本轮已对哪些目录拍过照」

def ensure_checkpoint(self, working_dir, reason="auto"):  # 每次改文件前调
    if not self.enabled: return False
    if working_dir in self._checkpointed_dirs: return False   # 本轮拍过 → 跳过
    self._checkpointed_dirs.add(working_dir)
    return self._take(working_dir, reason)                    # 真正拍照

逻辑很清晰:每轮对同一个目录最多拍一张快照。_checkpointed_dirs 这个集合就是「去重」的实现——new_turn() 清空它,ensure_checkpoint() 拍过就记下、再调直接跳过。这正是主循环第 1 步那行 new_turn() 的全部意义。

还有几个安全跳过:git 没装、目录是 / 或 home(太宽泛,不给整个家目录拍照)、未启用——都直接返回。而且 ensure_checkpoint() 永不抛异常(出错只静默记日志),保证它绝不拖垮主流程。

存储:一个共享的「影子 git 仓库」(最精妙处)

快照存在 ~/.hermes/checkpoints/store/,是一个所有项目共享的裸 git 仓库:

~/.hermes/checkpoints/store/
    objects/                  ← git 对象库(所有项目共享,内容去重)
    refs/hermes/<hash16>      ← 每个项目一个分支指针
    indexes/<hash16>          ← 每个项目一个 git 索引
    projects/<hash16>.json    ← {工作目录, 创建时间, 最后触碰}
  • 不碰你项目的 .git:通过设 GIT_DIR(指向影子库)+ GIT_WORK_TREE(指向你的项目目录)+ GIT_INDEX_FILE(项目专属索引)三个环境变量,让 git「在别处」记录你工作目录的快照。你项目自己的 git 状态完全不受影响。
  • 为什么用单一共享库:旧版每个目录一个独立影子库,同一个 repo 开十几个 worktree 就重复存十几份相同文件(~500MB)。改成共享库后,git 的内容寻址(相同内容只存一个 blob)能跨项目、跨轮次去重——新增一个 worktree 几乎零成本。

回滚:还能「撤销撤销」

def restore(self, working_dir, commit_hash, file_path=None):
    _validate_commit_hash(commit_hash)                 # 安全校验(见下)
    self._take(abs_dir, "pre-rollback snapshot ...")   # ← 先给当前状态再拍一张!
    git checkout <commit_hash> -- <目标>                # 再恢复到指定快照

注意那个 「pre-rollback snapshot」:回滚之前,它先把当前状态也拍一张。这样万一你回滚错了,还能再回滚回来——撤销也能撤销。可以整目录回滚,也可以只回滚单个文件(传 file_path)。

安全设计(防 git 参数注入)

  • _validate_commit_hash:只接受 4–40 位十六进制,并拒绝以 - 开头的值——否则会被 git 当成命令行参数,造成参数注入。
  • _validate_file_path:校验文件路径不越界。
  • 原因:commit hash / 路径可能来自不可信输入,必须防着被构造成 git flag。

自动维护(防膨胀)

影子库会越积越大,prune_checkpoints 负责清理:孤儿(工作目录已不存在)→ 删;陈旧(最后触碰超过 retention_days)→ 删;再 git gc --prune=now 回收;最后按 max_total_size_mb(默认 500)删最旧的。配置项还有 max_snapshots(每目录最多 20)、max_file_size_mb(单文件超 10MB 跳过)。

迭代预算:循环的安全阀

IterationBudget 本身很简单,就是一个线程安全的计数器:

# agent/iteration_budget.py
class IterationBudget:
    def __init__(self, max_total):
        self.max_total = max_total
        self._used = 0
        self._lock = threading.Lock()

    def consume(self) -> bool:        # 试消费一次,超额返回 False
        with self._lock:
            if self._used >= self.max_total:
                return False
            self._used += 1
            return True

    def refund(self):                 # 退还一次(execute_code 用)
        with self._lock:
            if self._used > 0:
                self._used -= 1

    @property
    def remaining(self):
        return max(0, self.max_total - self._used)
agent/iteration_budget.py
额度算的是「次数」,不是 token,也不是钱。 1 个额度 = 1 圈循环 ≈ 1 次调大模型(一圈 = 调模型 → 可能跑工具)。无论这一圈的请求是几万 token 还是几十 token,都只扣 1。它的目的是防止模型陷入死循环无限调用,而不是控制花费——所以用「次数」封顶最直接:跑满就强制停。

token 和成本是另一套独立的账,不在这里:它们在收尾的结果字典里(input_tokens、total_tokens、estimated_cost_usd 等,见第 5 篇),每次真实 API 调用后累加。迭代额度数圈数防死循环;token/成本计量数钱给你看账单——两条平行的账。
角色上限来源默认值
父 agentmax_iterations90
子 agentdelegation.max_iterations50

因为每个子 agent 有独立预算,所以「父 + 所有子」的总迭代数可以超过父的 90——这是有意的设计,让委派出去的子任务有自己的预算空间。

refund 的用处:execute_code(在沙箱里编程式地调工具)的那一圈会被退还,不计入预算。否则一次 execute_code 里跑了 20 个工具,会把额度瞬间烧光。

grace 调用:一个「预留的钩子」

循环条件末尾那个 or agent._budget_grace_call 是一个宽限机制:即使正常闸门(次数、预算)都关了,只要这个标志为真,就再放一圈。循环里也确实会在用掉后把它清成 False(agent/conversation_loop.py 约 631 行),保证「宽限」只生效一次。

准确说明(别被名字误导):在当前核心代码里,_budget_grace_call 只被初始化/清除为 False,没有任何地方把它置成 True。也就是说,预算用尽时不会自动触发 grace——循环会直接以 budget_exhausted 退出。这个标志是一个预留的扩展点:它被完整地「接线」进了循环条件(检查 + 用后清除),留给外部调用方在调 run_conversation 前主动设 True,以「格外多给一圈」。核心自身并不使用它。

循环为何退出:_turn_exit_reason

循环用一个诊断变量记录「为什么停」,排查问题时很有用。它初始为 unknown,然后被走到的那条退出分支覆盖成具体原因。常见取值:

值含义
text_response(finish_reason=stop)正常完成 —— 模型返回纯文本、不再要工具,拿到最终答复正常退出(最常见的结局)
interrupted_by_user用户中断(新消息 / /stop)
budget_exhausted迭代预算用尽(跑满圈数)
empty_response_exhausted模型反复返回空内容,重试用尽
all_retries_exhausted_no_response各种重试都用光仍没拿到响应
guardrail_halt被工具守卫策略叫停
ollama_runtime_context_too_smallOllama 运行时上下文不够装工具
unknown仅初始值 —— 若退出时仍是它,说明走了某条没记录原因的边缘路径(正常/异常退出都会覆盖成上面的具体值)
纠正一个易错点:unknown 不代表「正常完成」。正常拿到最终答复时,它会被设成 text_response(finish_reason=...)(源码约 4815 行)。unknown 只是没被任何分支覆盖时的兜底初始值。表里只列了常见值,源码里还有 interrupted_during_api_call、partial_stream_recovery、fallback_prior_turn_content 等更细的诊断值。

本篇要点

  • 循环由三个闸门守:调用次数、迭代预算、grace 标志。
  • 中断检查在每圈最开头,保证 /stop 及时生效。
  • 迭代预算是线程安全计数器,可 consume/refund;父 90、子 50,各自独立。
  • execute_code 的迭代会被退还,不占预算。
  • 预算耗尽时循环直接以 budget_exhausted 退出;grace 标志是预留扩展点,核心不主动触发。
