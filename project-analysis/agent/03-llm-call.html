调用模型

循环每一圈的核心动作:把消息发给模型、拿回响应。但「发请求」远不止一行 create()——发之前要对消息做大量清洗,发之后要处理重试与故障转移。这一篇就拆这两件事。

整体形状:组装一次,重试多次

effective_system = active_system_prompt + ephemeral...    # 组装系统提示
api_messages = [{"role":"system",...}] + history          # 组装消息(每圈一份新拷贝)
# … 一连串清洗(见下)…

while retry_count < max_retries:                           # 重试/故障转移循环
    api_kwargs = agent._build_api_kwargs(api_messages)
    response = client.chat.completions.create(**api_kwargs) # 真正发请求
    # 处理 finish_reason / 异常 / 故障转移
agent/conversation_loop.py · run_conversation() 循环体内
关键观念:api_messages 是本圈专用的临时拷贝,所有清洗都只改它,绝不动持久化的 messages / 会话历史。这是缓存铁律和「历史不可变」的实现手段。

发请求前的消息清洗

这是本篇的精华。每一圈在调模型前,要对 api_messages 做一长串处理,很多都是为了缓存命中和兼容不同模型:

  1. 拼接系统提示 + 临时系统提示

    effective_system = active_system_prompt + ephemeral_system_prompt

    缓存的 system prompt 放最前。若有「临时系统提示」(ephemeral),拼在后面——它只在本次 API 调用存在,不进历史。

  2. (可选)MoA 聚合 / prefill 注入

    若开了多模型聚合(MoA),把参考模型的产出聚合后拼进最后一条 user 消息;若有 prefill 消息,插在系统提示之后、历史之前。都是「API 调用时才有,不持久化」。

  3. Anthropic 缓存断点

    apply_anthropic_cache_control(api_messages, ...)

    对 Claude 模型(原生 Anthropic / OpenRouter / 兼容网关),注入 cache_control 断点(系统提示 + 最后 3 条消息),多轮对话可省约 75% 输入 token 成本。

  4. 清理孤儿工具结果

    agent._sanitize_api_messages(api_messages)

    无条件运行:补齐缺失的工具结果、剥掉没有对应调用的孤儿 tool 消息。防止会话加载或手工改消息后留下不合法结构,被 API 拒绝。

  5. 丢弃「只有思考」的助手轮

    agent._drop_thinking_only_and_merge_users(api_messages)

    有 reasoning 但既无可见输出也无 tool_calls 的助手轮会被丢掉,相邻 user 消息合并。防止 Anthropic 报「assistant 消息最后一块不能是 thinking」这类 400。只改本圈拷贝,历史里仍保留 reasoning 供 UI 显示。

  6. 规范化空白与 tool_call JSON(KV 缓存关键)

    把每条 content strip();把 tool_call 的 arguments 重新 json.dumps(sort_keys=True, 紧凑分隔符)。目的:让每一轮的前缀逐字节一致,从而命中本地推理服务器(llama.cpp / vLLM / Ollama)的 KV 缓存,也提高云端缓存命中率。

  7. 剥代理字符

    _sanitize_messages_surrogates(api_messages)

    再次剥掉孤立 surrogate,防止 OpenAI SDK 内部 json.dumps() 崩溃(省去 3 次重试)。

  8. 估算请求大小 + Ollama 上下文预检

    粗估 token 数用于日志;若是 Ollama 且预估请求超过运行时上下文,直接给出友好错误、退还本圈预算、跳出循环(_turn_exit_reason="ollama_runtime_context_too_small")。

这一串清洗解释了「为什么改一处历史就破坏缓存」:整套机制都建立在前缀逐字节稳定之上。任何中途改动历史的操作,都会让后面所有轮的前缀错位,缓存全部失效。

重试与故障转移循环

清洗完,进入内层 while retry_count < max_retries。它不只是「失败重发」,还负责跨提供商故障转移:

  1. 限流守卫(以 Nous 为例)

    agent.nous_rate_guard.nous_rate_limit_remaining()

    若别的会话已记录该提供商正在限流,直接跳过这次 API 调用(每次尝试都会加深限流)。能转移就 _try_activate_fallback() 切到备用提供商并重置重试计数;没有备用就落库并返回友好提示。

  2. 为当前提供商重塑 reasoning 字段

    agent._reapply_reasoning_echo_for_provider(api_messages)

    api_messages 是按主提供商组装的;中途转移到要求 reasoning_content 的提供商(DeepSeek/Kimi/MiMo)时,在这里幂等地补上,避免发出主提供商形状的陈旧字段。

  3. 构建 api_kwargs + 中间件 + 钩子

    agent._build_api_kwargs() → apply_llm_request_middleware() → pre_api_request 钩子

    把消息打包成 SDK 入参;过一遍 LLM 请求中间件(可改 payload、留 trace);触发插件的 pre_api_request 钩子。

  4. 真正发请求

    client.chat.completions.create(**api_kwargs)

    拿回 response 和 finish_reason。流式模式下经 stream_callback 逐块回吐(TTS 管线据此提前合成语音)。

  5. 按 finish_reason / 异常分流

    正常 stop → 交给工具/收尾逻辑;length 截断 → 续写重试;空内容 / 非法 JSON / 截断的 tool_call → 各自的重试计数与策略;网络/限流异常 → 重试或故障转移。

重试状态对象

这一堆重试不是用零散变量管的,而是集中在 TurnRetryState 和几个 agent._xxx_retries 计数(在准备阶段已归零):

计数针对
_invalid_tool_retries模型调了不存在的工具
_invalid_json_retriestool_call 参数不是合法 JSON
_empty_content_retries模型返回空内容
length_continue_retries响应被 length 截断,续写
compression_attempts循环内因超限触发的压缩次数

本篇要点

  • 消息组装一次、重试多次;api_messages 是临时拷贝,清洗只改它。
  • 发请求前的清洗大多为缓存命中(前缀逐字节稳定)和跨模型兼容而做。
  • Anthropic 缓存断点可省约 75% 输入成本。
  • 内层循环不只是重试,还承担跨提供商故障转移(限流守卫 + fallback)。
  • 各种异常各有独立重试计数,准备阶段统一归零。
