调用模型

循环每一圈的核心动作:把消息发给模型、拿回响应。但「发请求」远不止一行 create()——发之前要对消息做大量清洗,发之后要处理重试与故障转移。这一篇就拆这两件事。

整体形状:组装一次,重试多次

effective_system = active_system_prompt + ephemeral...    # 组装系统提示
api_messages = [{"role":"system",...}] + history          # 组装消息(每圈一份新拷贝)
# … 一连串清洗(见下)…

while retry_count < max_retries:                           # 重试/故障转移循环
    api_kwargs = agent._build_api_kwargs(api_messages)
    response = client.chat.completions.create(**api_kwargs) # 真正发请求
    # 处理 finish_reason / 异常 / 故障转移
agent/conversation_loop.py · run_conversation() 循环体内
关键观念:api_messages 是本圈专用的临时拷贝,所有清洗都只改它,绝不动持久化的 messages / 会话历史。这是缓存铁律和「历史不可变」的实现手段。

发请求前的消息清洗

这是本篇的精华,也是最容易被低估的部分。每一圈在调模型前,要对消息做十几步处理。它们大致分成 4 个阶段:先在原始 messages 上做防御性修复,再拷贝出本圈专用的 api_messages 并注入参考资料,然后组装系统提示与缓存断点,最后清洗规范化 + 大小预检。绝大多数步骤只有两个目的:命中缓存 和 兼容各家模型的怪脾气。

阶段 0 · 在原始 messages 上做防御性修复

这两步直接改 messages(不是拷贝),因为它们修的是「历史本身的结构损坏」。

  1. 修复损坏的 tool_call 参数

    agent._sanitize_tool_call_arguments(messages)

    历史里可能存着畸形的 tool_call arguments(截断的 JSON、流中断留下的残片)。这里先修好,免得发出去被 API 拒。修了几个会记日志。

  2. 修复消息角色交替

    repair_message_sequence_with_cursor(agent, messages)

    历史可能被卡成 tool → user 或 user → user 这种非法尾巴(比如空响应脚手架被剥掉后,新 user 消息落在了孤儿 tool 结果之后)。多数提供商对畸形序列会返回空内容,进而触发无限的「空响应重试」。这里主动修好角色交替;它还会重算 SessionDB 的落库游标,保证轮末刷库不会漏掉 assistant/tool 链(#44837)。

深入:这两步具体怎么修

A. 角色交替怎么修(repair_message_sequence)

它不是重排整个历史,而是做两趟扫描,只补两种最常见的破损:

# 第 1 趟:丢弃「孤儿 tool 消息」
known_tool_ids = set()
for msg in messages:
    if role == "assistant":
        known_tool_ids = {tc.id for tc in msg.tool_calls}   # 刷新:这一批 tool_call 的 id
    elif role == "tool":
        if msg.tool_call_id in known_tool_ids: keep(msg)    # 对得上 → 保留
        else: repairs += 1                                  # 对不上 → 丢弃(孤儿)
    elif role == "user":
        known_tool_ids = set()   # user 一来,tool 结果段就结束了

# 第 2 趟:合并相邻的 user 消息
if 当前是 user and 上一条也是 user:
    上一条.content += "\n\n" + 当前.content    # 用换行拼起来,一个字都不丢
    repairs += 1
  • 第 1 趟——丢孤儿 tool:维护一个「当前 assistant 发出的 tool_call id 集合」。每遇到 tool 消息,就看它的 tool_call_id 在不在这个集合里;对不上说明它没有对应的调用(孤儿)→ 直接丢。遇到 user 就清空集合(一段工具结果到此为止)。
  • 第 2 趟——并相邻 user:两个 user 挨着,就把后一条的文字用 \n\n 拼到前一条上,一个字都不丢。只合并纯文本;多模态(图片/音频的 list 结构)不动,以免搞坏附件。
  • 有修改就原地写回 messages[:] = merged,让落库/返回/DB 刷新都看到修好的序列。
它故意「不修」什么:assistant(tool_calls) → tool → user 这种看起来像坏尾巴的,其实可能是合法的——上一轮正常完成后,用户抢在模型续写前插了句话(正常的对话打断)。所以这里不回退这种,交给专门的「空响应脚手架剥离器」按标志位处理。修复要保守,只动铁定错的。

B. 损坏的 tool_call 参数怎么修(_repair_tool_call_arguments)

像 GLM / Ollama 这些模型常吐出截断的 JSON、多余逗号、Python 的 None、字符串里夹着裸控制字符——API 代理会用 HTTP 400「invalid tool call arguments」拒掉。这个函数是一条逐级兜底的修复链,从最轻到最重依次尝试:

级别症状 → 修法
快速路径空 / 只有空白 → 直接给 {}
快速路径字面量 None(Python 风格)→ 归一成 {}
Pass 0字符串里夹裸控制字符(tab/换行,llama.cpp 常见)→ 用 json.loads(strict=False) 解析,再 json.dumps 重新序列化成合法 JSON。这是最常见的本地模型修复(#12068)
Pass 1–3常规 JSON 修补:去掉 }/] 前的多余逗号;补齐没闭合的 {/[;删掉多余的闭合符号(最多迭代 50 次)
Pass 4转义字符串内的非法控制字符后再解析(应对同时有多种破损的情况)
最后兜底全都修不好 → 返回 {},让请求成功而不是让整个会话崩掉
这条修复链的哲学是「宁可丢参数,不可崩会话」:所有修复都记 WARNING 日志(方便排查),但即使一个参数彻底救不回来,也用空对象 {} 让这次工具调用「带着空参数」发出去——总好过整轮对话因为一段坏 JSON 直接挂掉。

阶段 1 · 拷贝出 api_messages 并注入参考资料

阶段 0 改的是原始 messages(修历史结构);从这里开始,一切只改临时拷贝 api_messages,把每条消息「塑造成当前提供商要的形状」,绝不动历史。

  1. 逐条拷贝 messages → api_messages

    api_messages = [msg.copy() for msg in messages]

    本圈专用的临时拷贝。后面所有清洗都只改它,原始 messages 分毫不动——这是「历史不可变 / 缓存不破」的根基。

  2. 向本轮 user 消息注入「参考资料」(机制 A)

    idx == current_turn_user_idx 时拼接

    把外部记忆预取(_ext_prefetch_cache → build_memory_context_block)和插件 pre_llm_call 上下文(_plugin_user_context)拼到本轮 user 消息后面。只改这次调用的副本,不写回历史(所以第 1 篇讲的「干净原文」得以保留)。注意:注入的是参考资料,不是提醒。

  3. 回填 reasoning 字段(适配各家格式)

    agent._copy_reasoning_content_for_api(msg, api_msg)

    推理模型(o1、DeepSeek-R1、Claude thinking、Kimi 等)在给答案前会先产出一段思考过程(reasoning)。Hermes 用统一格式把它存在历史里,但各家提供商要求的形状不同:有的要内联在 content 里的 <think>…</think> 标签,有的(Moonshot / Kimi、DeepSeek)要放在独立的 reasoning_content 字段,Anthropic 用带签名的 thinking 块。这一步就是把历史里存的 reasoning,按当前提供商要的形状回填到副本 api_msg 上——否则格式对不上会被拒(比如 Moonshot 要求「带 tool_calls 的 assistant 消息必须有 reasoning_content」)。只改副本,历史里的统一存法不动。

reasoning 会持久化到数据库。 SessionDB 的 messages 表专门留了四列存它:reasoning(统一存法)、reasoning_content(独立字段形状)、reasoning_details(Anthropic 带签名的 thinking 块)、codex_reasoning_items(Codex)。存它是为了:UI 展示(重开会话还能看它怎么想的)、多轮回放(有的提供商强制要求回放上轮 reasoning,如 Anthropic 签名、Codex replay)、断点续聊。
关键:库里存「统一版」,发请求时才「变形」成当前提供商的形状(只改副本);阶段 3 的「丢弃只有思考的助手轮」也只从 API 副本丢,库里的 reasoning 照样保留。

阶段 2 · 组装系统提示与额外内容

  1. 拼接系统提示 + 临时系统提示,放最前

    effective_system = active_system_prompt + ephemeral_system_prompt

    缓存的 system prompt 放请求最前。若有「临时系统提示」(ephemeral)拼在后面。作为单一 content 字符串发送,保证跨轮字节稳定、上游缓存持续命中。

  2. (可选)MoA 多模型聚合注入

    aggregate_moa_context(...) → 拼进最后一条 user 消息

    MoA(Mixture of Agents,多智能体混合)相当于给主模型配一个「顾问团」:开启后,先拿当前对话去并行问几个参考模型(reference_models,线程池并发),各自给出看法;再由一个聚合模型(aggregator)把这些看法综合成一段简明可行的建议(聚焦下一步、工具策略、风险、分歧点)。这段综合被包成 [Mixture of Agents context …] 块,拼到最后一条 user 消息后面,当作给主模型的私有参考(明确告诉聚合器:别直接答用户,只产出主 agent 循环要用的上下文)。

    要点:参考模型并发调用、顺序对齐;某个参考失败不会中断主循环——它变成一条带标签的「失败备注」,主模型带着部分上下文照样干。默认不限 max_tokens(让每个模型用自己的上限,避免长综合被截断)。整段只在本次调用拼接,不持久化。

  3. (可选)prefill 消息注入

    插在系统提示之后、历史之前

    agent.prefill_messages 是一组预置消息,若有,则插到系统提示之后、对话历史之前(sys_offset = 1 if 首条是 system else 0,依次 insert)。用途:给对话「垫」一些前置内容——比如 few-shot 示例、强制的助手开场前缀等。同样是API 调用时才有,不写回历史/不持久化。

  4. Anthropic 缓存断点

    if agent._use_prompt_caching: apply_anthropic_cache_control(api_messages, ...)

    这是 Claude 特有的缓存机制。区别于 OpenAI/DeepSeek/Kimi 的自动服务端前缀缓存,Anthropic 不自动缓存——你必须在请求里手动打「缓存到这里为止」的标记(cache_control),它才缓存。

    「缓存断点」是什么:在某个内容块上打一个 cache_control: {"type":"ephemeral"} 标记,含义是「从请求最开头一直缓存到这里」。下次请求若这段前缀逐字节相同就命中缓存,不重算。Anthropic 每个请求最多 4 个断点。

    Hermes 打在哪:系统提示 + 最后 3 条消息。系统提示是最前、最大、每轮不变的前缀(缓存收益最大);末尾几条则让「不断变长的历史」也被增量缓存(这轮末尾内容,下轮就成了可复用前缀)。

    为什么省约 75%:Anthropic 缓存命中部分的读取价格只有正常输入的约 1/10(首次写入略贵一点,后续读很便宜)。多轮对话里请求的绝大部分都是可缓存前缀,整体输入成本降到约 1/4。

    具体怎么加(apply_anthropic_cache_control):

    # 1. 深拷贝(不动原件,因为要改嵌套结构)
messages = copy.deepcopy(api_messages)
# 2. 造标记:默认 5 分钟存活;1h 则加 ttl
marker = {"type": "ephemeral"}          # cache_ttl=="1h" 时再加 "ttl":"1h"
# 3. 放最多 4 个断点(策略名 system_and_3)
if messages[0].role == "system": mark(messages[0])   # 断点①:系统提示
for m in 最后 (4-已用) 条非 system 消息: mark(m)        # 其余断点:末尾几条

    标记怎么「贴」上去(_apply_cache_marker,要处理不同 content 形态)——关键:Anthropic 的 cache_control 是挂在内容块上,不是挂在消息对象上:

    • content 是字符串 → 先包成内容块:content = [{"type":"text","text":原文,"cache_control":marker}](纯字符串没法直接挂,必须转成块)
    • content 是列表(已是多块)→ 把标记挂到最后一个块上
    • content 为空 → 直接挂到消息对象上
    • tool 消息 → 只在原生 Anthropic 时标,OpenRouter/兼容网关跳过

阶段 3 · 清洗、规范化、大小预检

  1. 清理孤儿工具结果

    agent._sanitize_api_messages(api_messages)

    无条件运行:补齐缺失的工具结果、剥掉没有对应调用的孤儿 tool 消息。防止会话加载或手工改消息后留下不合法结构被 API 拒。

    怎么实现的(集合差集配对):

    # ① 角色白名单:role 不在 {system,user,assistant,tool} 的消息直接丢
# ② 算两个集合
surviving_call_ids = {所有 assistant.tool_calls 的 id}   # 发起了哪些调用
result_call_ids    = {所有 tool.tool_call_id}            # 存在哪些结果
# ③ 两个差集分别处理
孤儿结果 = result_call_ids - surviving_call_ids   # 有结果没调用 → 删掉这些 tool 消息
缺失结果 = surviving_call_ids - result_call_ids   # 有调用没结果 → 补占位 tool 消息

    对「缺失结果」,会在对应 assistant 后插一条占位:{"role":"tool", "content":"[Result unavailable — see context summary above]", "tool_call_id":cid}。

    核心契约:Chat API 要求每个 tool_call 必须有配对的 tool 结果、每个 tool 结果也必须有配对的调用,缺一不可。所以:多出来的结果删掉,缺失的结果补桩。
    为什么缺失要补桩而不是删调用:tool_call 嵌在 assistant 消息里、不好单独摘除,所以反过来给它补一个占位结果,让「调用→结果」这对补全(常见于压缩把结果删了却留着调用)。
    与阶段 0 的区别:阶段 0 的 repair_message_sequence 管的是「角色交替」(改原始 messages);这一步管的是「调用/结果配对完整性」(改 api_messages 副本),两者互补。

  2. 丢弃「只有思考」的助手轮

    agent._drop_thinking_only_and_merge_users(api_messages)

    「只有思考的助手轮」= 一整条 assistant 消息有 reasoning、但既无可见 content 也无 tool_calls(只想了想、啥也没说没做)。Anthropic 会拒绝「最后一块是 thinking」的 assistant 消息(400)。这一步把它们从 API 副本里剔除。

    怎么实现(两趟):

    # 第 1 趟:丢掉「只有思考」的 assistant 轮
kept = [m for m in messages if not _is_thinking_only_assistant(m)]
# 第 2 趟:丢完可能让两个 user 挨到一起 → 合并它们(保住角色交替)
if prev.role == "user" and m.role == "user":
    prev.content += "\n\n" + m.content   # 各种 content 形态(str/list)都处理
    为什么是「丢+并」而不是「补一句占位」:如果给它塞个 "." 或「(继续)」当占位,等于在历史里伪造了模型没说过的话,后续轮会当真。丢掉更诚实;而丢掉夹在两个 user 之间的 assistant 会留下 user→user,所以再合并相邻 user 保住角色交替。这套做法照搬自 Claude Code 的 normalizeMessagesForAPI。只改副本,历史里的 reasoning 照样保留供 UI 显示。

  3. 规范化空白与 tool_call JSON(KV 缓存关键)

    把每条 content strip();把 tool_call 的 arguments 重新 json.dumps(sort_keys=True, 紧凑分隔符)。若 arguments 无法解析,则调 _repair_tool_call_arguments 兜底修复。这一步是整篇「缓存」主题的技术核心,值得讲透。

    先理解:缓存是按「字节」精确匹配的。无论本地推理服务器的 KV 缓存还是云端的前缀缓存,缓的都是请求前缀的计算结果。下次请求来,把新前缀和缓存里的逐字节比对——只要差一个字符(哪怕一个空格),就判定「不同前缀」,缓存不命中,从那个字符起全部重算。

    多轮对话本该命中:第 2 轮请求 = 第 1 轮 + 末尾新增,所以第 2 轮的前缀(system + 第 1 轮历史)本该和第 1 轮发出去的一模一样 → 命中缓存 → 跳过整段重算,又快又省。

    问题:同一条消息可能被序列化成不同字节——意思一样、字节不同:

    同一内容写法 A写法 B
    尾部空格"hello""hello "
    JSON 键顺序{"a":1,"b":2}{"b":2,"a":1}
    JSON 空格{"a":1}{"a": 1}

    只要第 1 轮和第 2 轮对同一个 tool_call 序列化得稍有不同,前缀字节就变了——明明啥都没变,缓存却全 miss。

    解法:把序列化「归一化」,让同样的内容永远产出同样的字节:

    • content.strip() —— 消除首尾空白差异。
    • json.dumps(arguments, sort_keys=True, separators=(",",":")) —— 键永远排序、无多余空格,同一组参数每轮都序列化成完全相同的字节。

    为什么这两个参数就能保证字节相同?因为对同一份数据,JSON 序列化只有两个「自由度」会让输出变化,这两个参数正好各锁死一个:

    自由度参数作用
    ① 键的顺序(dict 按插入顺序保存)sort_keys=True序列化前先把键按字母排序,消除顺序差异
    ② 分隔符的空格(默认 ", "/": " 带空格)separators=(",",":")用不带空格的紧凑分隔符,消除空格差异
    a = {"path": "a.py", "line": 5}
b = {"line": 5, "path": "a.py"}     # 内容相同,顺序不同

# 默认:字节不同 ❌
json.dumps(a)  # '{"path": "a.py", "line": 5}'
json.dumps(b)  # '{"line": 5, "path": "a.py"}'

# 加 sort_keys + 紧凑分隔符:字节完全相同 ✅
json.dumps(a, sort_keys=True, separators=(",",":"))  # '{"line":5,"path":"a.py"}'
json.dumps(b, sort_keys=True, separators=(",",":"))  # '{"line":5,"path":"a.py"}'  ← 一模一样

    JSON 里对同一份数据能变的基本就这两处,全锁死后输出就成了唯一的「规范形式」——不管这份 arguments 是模型重新吐的、被解析后重建的、还是走了不同代码路径,只要内容一样,字节就一样。

    为什么还要「确定性 call_id」:有些提供商/本地模型返回 tool_call 时不给 id,Hermes 得自己生成一个。如果每轮处理到这条消息都生成一个随机 UUID,那这条消息的字节每轮都不一样 → 前缀错位 → 直接废掉前缀缓存。所以 Hermes 用 _deterministic_call_id(拿函数名+参数+序号算出来),保证同一个调用每轮都得到同一个 id,字节稳定、缓存能命中。
    情况典型来源
    给 id(用不上 fallback)OpenAI Chat Completions(call_abc123)、Anthropic Messages(toolu_xxx)、多数云端兼容服务(GPT/Claude/DeepSeek 云端)
    缺 id 需生成代码里明确的是 OpenAI Responses API / Codex(响应结构不同,用 fc_… item id);实际最常触发的是本地模型(Ollama / llama.cpp 上的 GLM、Kimi、Qwen 等,只吐 JSON 不附标准 id)
    # 生成方式(agent/codex_responses_adapter.py)
seed = f"{fn_name}:{arguments}:{index}"           # 函数名+参数+序号
return "call_" + sha256(seed)[:12]                # 例:call_a1b2c3d4e5f6 —— 同调用永远同 id
    一句话:凡是「同样内容却可能产出不同字节」的地方(空格、键顺序、随机 id),都要归一化,否则前缀一变、缓存全失效。这就是「缓存铁律」在这一步的具体落地。

  4. 剥代理字符

    _sanitize_messages_surrogates(api_messages)

    继准备阶段之后的第二道防线,原地遍历所有消息,把孤立 surrogate 替换成 \ufffd(Unicode 替换字符 �),防止 OpenAI SDK 内部 json.dumps() 崩溃(省去 3 次重试)。

    覆盖面很全:不只查 content,还查 name、tool_calls 的 id/function.name/arguments,以及 reasoning/reasoning_content/reasoning_details 等附加字段。因为字节级推理模型(mimo、kimi、glm)常在 reasoning 里吐出孤立 surrogate,这些会流进下一轮的 api_messages 再引爆 json.dumps——所以每个可能藏字符串的字段都得扫。
    具体例子:😀 是怎么变成「半个非法字符」的 
    # 😀(U+1F600)在 UTF-16 下要用「一对」surrogate 表示:
😀 = U+D83D(高位) + U+DE00(低位)      # 成对才合法

# 模型流式吐 😀 时,吐完前半个就被截断 → 只剩「半个」:
content = "你好\ud83d世界"              # \ud83d = 半个 😀,孤立高位,非法

# OpenAI SDK 内部 json.dumps 一碰就崩:
json.dumps({"content": "你好\ud83d世界"})
# → UnicodeEncodeError: 'utf-8' codec can't encode character '\ud83d': surrogates not allowed

# Hermes 的修法:替换成 �(U+FFFD,合法),保留「这里原有字符」的位置提示
"你好\ud83d世界"  →  "你好\ufffd世界"  →  "你好�世界"
    为什么本地模型特别容易出:mimo/kimi/glm 是字节级模型,一个 emoji/生僻字要连吐好几个字节,在流式或 reasoning 输出时容易在「多字节字符没吐完整」时被切断,留下孤立 surrogate。

  5. 估算请求大小 + Ollama 上下文预检

    estimate_request_tokens_rough(...) → _ollama_context_limit_error(...)

    粗估字符数与 token 数(total_chars、approx_tokens、approx_request_tokens)用于日志。然后专门给 Ollama 做一道预检:

    # 仅当有工具 且 Ollama 运行时上下文 小于最低要求 时触发
if agent.tools and 0 < _ollama_num_ctx < MINIMUM_CONTEXT_LENGTH:
    return 友好错误("Ollama 只加载了 N tokens 上下文,Hermes 工具调用至少需要 M。
                    建议设 model.ollama_num_ctx: 65536 后重载模型")

    为什么专门管 Ollama:本地跑 Ollama 时,模型的运行时上下文是加载时定死的,若太小(装不下 system prompt + 工具 schema),工具调用会各种出错。与其让它发出去反复失败,不如提前拦下、给出「怎么调大」的明确指引、退还本圈预算、跳出循环(_turn_exit_reason="ollama_runtime_context_too_small")。只在有工具时才检查(纯聊天不吃这个约束)。

这一长串清洗解释了「为什么改一处历史就破坏缓存」:整套机制都建立在前缀逐字节稳定之上。阶段 0 修的是历史结构(改 messages),阶段 1–3 全在临时拷贝 api_messages 上做、绝不动历史。任何中途改动历史的操作,都会让后面所有轮的前缀错位,缓存全部失效。

重试与故障转移循环

清洗完,进入内层 while retry_count < max_retries。它不只是「失败重发」,还负责跨提供商故障转移:

  1. 限流守卫(以 Nous 为例):跨会话的「共享断路器」

    agent/nous_rate_guard.py · nous_rate_limit_remaining()

    要解决的问题:Nous Portal 有 RPH(每小时请求数)限制,一旦被限流,每多发一次请求(连 SDK 内部重试都算)都会加深限流。更糟的是 Hermes 常同时跑多个会话(CLI、网关、cron 定时、辅助副 LLM);如果一个会话被限流、其他会话还在猛敲 Nous,大家一起把坑挖得更深。

    解法:一个所有会话共享的状态文件(~/.hermes/…/nous.json)。谁撞上限流,就把「解除时间」写进这个文件;所有会话发请求前先读它,发现还在限流期就根本不发。

    # 三个函数,围绕一个共享文件 nous.json:
record_nous_rate_limit(headers)   # 撞限流 → 原子写入 {"reset_at": 解除时间戳}
                                  #          (temp 文件 + rename,防并发读到半截)
nous_rate_limit_remaining()       # 读文件:reset_at - now > 0 ? 返回剩余秒数 : None(顺手删过期文件)
clear_nous_rate_limit()           # 成功请求后删文件(证明限流已解除,大家可恢复)

    守卫怎么用它(在重试循环最顶上、发请求之前):

    if agent.provider == "nous":
    remaining = nous_rate_limit_remaining()        # 读共享文件
    if remaining and remaining > 0:                # 别的会话已记录:Nous 还在限流
        if agent._try_activate_fallback():          # 能转移 → 切备用提供商
            retry_count = 0; compression_attempts = 0
            continue                                #   跳过 Nous,用备用重来
        # 没备用 → 落库,返回友好提示(告诉你还有多久解除)
        return {"final_response": f"⏳ Nous 限流中,{format_remaining(remaining)} 后解除…",
                "failed": True, ...}
    关键点:这是个跨进程/跨会话的断路器——A 会话撞到限流写文件,B、C 会话(可能是网关、定时任务)读到后主动避让,不去白白加深限流。
    还要辨真假:Nous Portal 背后多路复用了 DeepSeek/Kimi/MiMo/Hermes 等上游,一个 429 可能只是某个上游临时没容量(几秒就好),而非你的账号真被限流。所以 is_genuine_nous_rate_limit 会用响应头的 x-ratelimit-* + 上次成功状态来判断,只有确认是账号级限流才写这个跨会话文件——否则一个上游抖动会误锁所有会话几分钟。
    转移时别忘了「换名牌」:切到备用提供商后,守卫紧接着调 _sync_failover_system_message(agent, api_messages, ...)。因为 _try_activate_fallback() 只改了 agent._cached_system_prompt 里的 Model:/Provider: 身份行,而本轮的 api_messages[0](系统消息)是转移前组装的、还印着旧提供商的名字。这一步就地把 api_messages[0]["content"] 刷成新身份(带上 ephemeral 补充),否则整轮(网关模式下每条消息都会重新激活备用)都会顶着过时的身份行发出去。同时 retry_count、compression_attempts 归零、primary_recovery_attempted=False——给备用提供商一次干净的重来。

  2. 为当前提供商重塑 reasoning 字段(双向:该补补、该删删)

    agent._reapply_reasoning_echo_for_provider(api_messages) → agent_runtime_helpers.reapply_reasoning_echo_for_provider

    api_messages 只在进重试循环之前按主提供商组装一次;循环里每圈都从这份 list 重新 _build_api_kwargs。可一旦中途故障转移换了提供商,那些烤进 api_messages 的 reasoning 字段就是按旧提供商的口味做的,得跟当前提供商对齐。函数开头一句 needs_pad = agent._needs_thinking_reasoning_pad(),然后遍历所有 assistant 轮,分两个方向处理:

    当前提供商needs_pad动作不做会怎样
    要求思考字段
    (DeepSeek / Kimi / MiMo 思考模式)    		True给缺 reasoning_content 的助手轮补上回显
    (copy_reasoning_content_for_api)    		400:The reasoning_content in the thinking mode must be passed back
    拒收思考字段的严格提供商
    (Mistral / Cerebras / Groq / SambaNova…)    		False把残留的 reasoning_content pad 删掉(pop)400/422:Extra inputs are not permitted

    两个容易忽略的细节:

    • strip(删)那条对应一个真实 bug #45655:DeepSeek 主提供商给历史垫了 " ",请求回退到 Mistral,Mistral 就对这个陈旧的 pad 报 422。所以「补」和「删」缺一不可。
    • 「补」的时候填什么,不是随便垫个空格——见下面的「取值瀑布」。
    「补」的内容从哪来:5 级取值瀑布

    agent_runtime_helpers.copy_reasoning_content_for_api(agent, source_msg, api_msg)

    补 reasoning_content 时不是凭空造、也不是无脑填空格。核心原则:优先用这一轮自己真实的思考;真思考缺失或跨了提供商,才退到一个空格 " " 占位;绝不伪造、也绝不搬运别家的思维链。按优先级依次判断:

    #这一轮的状态填什么为什么
    1已经有 reasoning_content 字符串原样保留(是空串 "" 则升级成 " ")那本来就是它自己的思考 / 占位;"" 会被 DeepSeek V4 Pro 400,得升成空格(#17341)
    2有 tool_calls + reasoning 字段,但那个 reasoning 是别的提供商写的一个空格 " "故意不把别家思维链拷过来——防泄漏(如把 MiniMax 的 CoT 泄给 DeepSeek/Kimi,#15748)
    3有自己的 reasoning 字段(同提供商)把 reasoning 提升成 reasoning_content这是真思考内容,直接搬过来
    4啥思考都没有一个空格 " "纯占位,满足「思考模式必须回传 reasoning_content」的硬性要求(#17341)
    5reasoning_content 存在但不是字符串(如压缩后变 None)删掉别把 null 发给 API
    两个刻意的取舍:
    ① 为什么是空格不是空串?DeepSeek V4 Pro 收紧了校验,"" 会 400(must be passed back)," " 才过(#17341)。
    ② 为什么不搬别家的思考?跨提供商会泄漏思维链,所以情形 2 宁可填空格也不拷 reasoning(#15748)。
    另外:每个方向也都受 needs_thinking_pad 门控——当前是严格提供商时,即便有内容也一律 pop 掉,和上面「双向」表的删除方向一致。
    「幂等」的准确含义:它把 api_messages 里的 reasoning 字段和当前提供商对齐——该补补、该删删、已经对上就啥也不做,返回「改了几轮」。因为每次都是照当前提供商重新裁决(而非在上次基础上叠加),放在循环里每圈都调都安全,且覆盖所有故障转移路径:无论从谁转到谁,发出去的 api_messages 形状永远对得上正在答话的那个提供商。

  3. 构建 api_kwargs + 中间件 + 钩子

    agent._build_api_kwargs() → apply_llm_request_middleware() → pre_api_request 钩子

    ① 组装 SDK 入参:_build_api_kwargs(转发到 chat_completion_helpers.build_api_kwargs)按 api_mode 分四条路打包成对应后端的入参:

    • anthropic_messages → Anthropic Messages API(带缓存断点、reasoning、OAuth 处理)
    • bedrock_converse → AWS Bedrock Converse(绕开 OpenAI 客户端,走 boto3)
    • codex_responses → OpenAI/xAI 的 /responses(xAI 这条还会深拷贝并剥掉工具 schema 里的 pattern/format/含斜杠的 enum,否则 400)
    • chat_completions(默认)→ 标准 OpenAI 兼容入参(model/messages/tools/max_tokens/reasoning…)

    ② 两个即时后处理:若 _force_ascii_payload 置位(之前撞过编码问题),_sanitize_structure_non_ascii(api_kwargs) 把非 ASCII 字符压平;codex_responses 模式再过一遍 preflight_kwargs() 做传输层预检。

    ③ LLM 请求中间件:apply_llm_request_middleware(api_kwargs, …) 返回三样东西——改写后的 payload(实际发出)、original_payload(改写前的原样,留证)、trace(每个中间件干了什么)。它带全上下文(task_id/turn_id/api_request_id/session/platform/model/provider…),可改 payload 也可记录。整个 try 兜底:中间件抛错就退回原始 api_kwargs、空 trace,绝不让它挡住请求。

    ④ pre_api_request 钩子:仅当有插件注册了该钩子(has_hook)才触发。给插件一份脱敏后的请求视图(request["body"]["messages"])+ 一堆元数据(消息数、工具数、约估 token、字符数、起始时间、中间件 trace…)。消息 list 浅拷贝一份传进去——防止插件异步快照时观察到后续对 api_messages 的改动,又不至于像深拷贝那样把每张 base64 图、每条工具结果都走一遍。

    调试彩蛋:置 HERMES_DUMP_REQUESTS=1,发请求前会把完整 api_kwargs 落盘(reason="preflight"),用来核对「到底发了什么给模型」。

  4. 真正发请求(附「响应形状校验」闸门)

    run_llm_execution_middleware(api_kwargs, _perform_api_call, …) → transport.validate_response(response)

    发送本身只是一个闭包 _perform_api_call:流式走 _interruptible_streaming_api_call(kwargs, on_first_delta=_stop_spinner)(第一个 delta 一到就停「思考中」动画),非流式走 _interruptible_api_call;整体再被 run_llm_execution_middleware 包一层(详见下面的「深入:请求到底怎么发出去的」)。流式模式下经 stream_callback 逐块回吐,TTS 管线据此提前合成语音。

    拿回 response 后,并不是马上就用——先记 api_duration、停动画、打印耗时,然后过一道响应形状校验闸门。因为「HTTP 200 但内容是空的/坏的」很常见(限流、上游超时、服务端 5xx、被截断的流),必须在往下走之前拦住:

    api_mode校验(各 transport 的 validate_response)
    chat_completions(默认)有 choices 属性、非 None、非空
    anthropic_messagescontent 是非空 list
    codex_responsesoutput 非空;status=failed/cancelled 判失败;但 output_text 有内容时留给归一化兜底
    bedrock_converse有 output 或 choices
    校验不过 = 急切故障转移(eager fallback):一旦判 response_invalid,先触发 _invoke_api_request_error_hook(reason="invalid_response", retryable=True)、静默停动画、retry_count += 1;然后不做指数退避,直接 _try_activate_fallback() 切备用提供商。原因写在源码注释里:空/坏响应是限流的常见征兆,与其原地退避重试,不如立刻换一家——退避只是白白多等。

  5. 按 finish_reason / 异常分流

    响应通过校验后,才进入归一化并取 finish_reason 分流:正常 stop → 交给工具/收尾逻辑;length 截断 → 续写重试;空内容 / 非法 JSON / 截断的 tool_call → 各自的重试计数与策略;网络/限流异常(走 except)→ 重试或故障转移(见下面的「异常与错误处理」)。

深入:请求到底怎么发出去的

「发请求」这一步远不是一句 create() 那么简单。真实路径里有三层包装,每一层都为了健壮性。

① 永远优先走「流式」——即使没人看流

你可能以为只有需要「实时打字」时才用流式,但 Hermes 默认总是走流式路径(_use_streaming = True),哪怕根本没有显示/TTS 消费者。

为什么?流式路径能做细粒度的健康检查——按「距上一次真实内容块过了多久」来判断僵流(流式默认 HERMES_STREAM_STALE_TIMEOUT=180s,非流式 HERMES_API_CALL_STALE_TIMEOUT=90s)。非流式路径没有这些。有些提供商会用 SSE 心跳把连接「假装活着」却永远不返回真正内容,非流式调用就会无限期挂死(子 agent、quiet 模式尤其容易中招)。走流式就能及时发现「这流卡住了」并中止。没有消费者时,流式的回调只是空转,不影响功能。下面展开讲。

几种例外会切回非流式:

  • _disable_streaming 已置位 —— 之前某次调用时提供商明确说「不支持流式」,本会话后续就不再每次都撞墙。
  • provider 是 copilot-acp / moa,或 base_url 是 acp:// 类 —— 它们经子进程 stdio 通信,返回的不是可迭代的流。
  • 测试里的 Mock 客户端 —— 返回的是 SimpleNamespace,不是流迭代器。

SSE 心跳「假活」与僵流检测器

病:OpenAI 兼容的流式接口用 SSE(Server-Sent Events)——服务端保持一条 HTTP 连接不关,随时推 data: {...} 数据块,直到 data: [DONE]。中间的代理/负载均衡器会把「一段时间没字节流动」的连接当死连接掰掉。为不被掰,很多提供商会往连接里塞心跳帧(keepalive):通常是 SSE 注释行(以 : 开头,如 OpenRouter 的 : OPENROUTER PROCESSING)或空 ping。这些心跳不含任何真实内容(没有 token delta),只是制造字节流动。于是当模型在上游卡死(过载/死锁/无限思考/排队)永远不吐 token时,代理还在发心跳——从最原始的 TCP 视角看连接完全健康,头脑简单的客户端就无限期干等。

药:僵流检测器(stale-stream watchdog)。关键在测什么——不问「socket 还活着吗」,而问「距离上一次收到真正的内容块过了多久」。心跳帧不会重置这个「真实内容」计时器——它们能骗过 TCP,但不算进展。窗口内没等到有意义的块,就判僵流、掉连接、换新连接重试(或故障转移)。

源码里的分类逻辑:上游流真断时,OpenAI SDK 会抛出没有 status_code 的 APIError(区别于带状态码的 4xx/5xx),Hermes 用一串短语(connection lost/reset/terminated、broken pipe、upstream connect error…)识别成「SSE 连接错误」,然后关旧 client、重建连接池、重试。

一个必要的例外:推理模型「合法地」久不吐字。推理模型(o1/o3、DeepSeek R1、Grok reasoning、Nemotron…)会先思考很久才吐第一个 token,这不是假活。若一刀切 180s 会把正在思考的模型误杀(表现为 BrokenPipeError/RemoteProtocolError)。所以有个按模型抬高下限的机制 get_reasoning_stale_timeout_floor——比如 Nemotron-3-Ultra / DeepSeek-R1 抬到 600s,用 max(默认值, floor),只抬不降、绝不覆盖用户显式配置(providers.<id>.models.<model>.stale_timeout_seconds)。

这也正是「即使没人看流也强制走流式」的深层动机:非流式只有一次阻塞读,提供商用心跳吊着连接就是不发最终 JSON body 时,它没有块级信号,只能干等到(粗得多的)整体超时——或永远。流式的每块可见性才让僵流检测成为可能。

② 调用是「可中断」的

真正发请求走 _interruptible_streaming_api_call()(或非流式的 _interruptible_api_call())。「可中断」意味着:请求发出去、还在等模型吐字的过程中,用户按 /stop 也能当场打断,而不用干等这次调用结束。

③ 外面再套一层执行中间件

整个调用被 run_llm_execution_middleware(api_kwargs, _perform_api_call, ...) 包住。中间件能观察/干预这次执行(记录、改写、注入 trace),把「准备好的参数」和「真正执行的函数」解耦。

_use_streaming = True                         # 默认流式
if agent._disable_streaming: _use_streaming = False
elif provider in {copilot-acp, moa}: _use_streaming = False
elif not has_stream_consumers() and isinstance(client, Mock): _use_streaming = False

def _perform_api_call(kwargs):
    return (agent._interruptible_streaming_api_call(kwargs, on_first_delta=_stop_spinner)
            if _use_streaming else agent._interruptible_api_call(kwargs))

response = run_llm_execution_middleware(api_kwargs, _perform_api_call, ...)  # 执行
api_duration = time.time() - api_start_time                                  # 计时

模式 A · 响应无效

核心思路:先转移、按错误码给人话、指数退避、随时可中断。

① 无效响应 → 优先「急切故障转移」

空/畸形响应往往是限流的症状。所以不急着原地重试,而是立刻尝试切到备用提供商:

retry_count += 1
if agent._try_activate_fallback():        # 有备用就立刻切
    active_system_prompt = _sync_failover_system_message(...)
    retry_count = 0; compression_attempts = 0   # 换了提供商,计数清零重来
    continue

切换后重试计数清零——因为是全新的提供商,不该背前一个的重试账。

② 没备用了 → 按错误码翻译成「人话」

不再一律当「限流」处理,而是从响应错误码 + 响应耗时,拼出一句可读的失败诊断:

信号诊断(_failure_hint)
错误码 524上游提供商超时(Cloudflare 524)
504上游网关超时
429被上游限流
500 / 502上游服务器错误
503 / 529上游过载
无错误码 & 耗时 < 10s响应太快 → 多半是被限流秒拒
无错误码 & 耗时 > 60s响应太慢 → 多半是上游超时

③ 重试到顶 → 最后再试一次备用,否则放弃

retry_count >= max_retries 时,先看还有没有 pending fallback 可切;实在没有,才落库并返回一个失败结果字典(failed=True、带 error 说明),而不是抛异常炸出去。

④ 退避:抖动指数退避,且随时可中断

wait_time = jittered_backoff(retry_count, base_delay=5.0, max_delay=120.0)
# 抖动指数退避:5s 起步,最高封顶 120s;加随机抖动避免「重试风暴」同步撞上游

sleep_end = time.time() + wait_time
while time.time() < sleep_end:
    if agent._interrupt_requested:        # 退避等待中也能被 /stop 打断
        close_interrupted_tool_sequence(messages, _interrupt_text)
        return {..., "interrupted": True}
    time.sleep(0.2)                       # 每 0.2s 醒一次
    if 累计到 ~30s: agent._touch_activity(...)   # 让网关知道「我还活着」
几个用心的细节:抖动(jitter)让多个会话不会在同一时刻齐刷刷重试、把上游再次打垮;0.2s 分片睡眠让长达 2 分钟的退避也能秒响应中断;退避期间每 30 秒 touch 一次活动时间,免得被网关的不活动监控误判为卡死。
「抖动指数退避」到底怎么算

agent/retry_utils.py · jittered_backoff(attempt, base_delay=5.0, max_delay=120.0, jitter_ratio=0.5)

名字拆成两半理解:指数退避是确定的「每次翻倍」,抖动才是那个随机数。

① 指数退避(不随机)——每失败一次等待翻倍,并封顶:

delay = min(base_delay × 2^(attempt-1), max_delay)
第几次重试计算基础等待
15 × 2⁰5s
25 × 2¹10s
35 × 2²20s
45 × 2³40s
55 × 2⁴80s
65 × 2⁵ = 160 → 封顶120s
7+…120s

越等越久,给过载/限流的上游更多恢复时间,别一失败就猛敲。这部分完全确定,没有随机。

② 抖动(这部分才是随机数)——在基础等待之上再加一个随机偏移:

jitter = random.uniform(0, jitter_ratio × delay)   # jitter_ratio = 0.5
最终等待 = delay + jitter                            # 落在 [delay, 1.5×delay]

比如第 3 次:基础 20s,抖动随机落在 [0, 10s] → 最终等 20~30s 之间的某个值。

为什么要加随机?防「惊群 / 重试风暴(thundering herd)」。假设 100 个会话在同一瞬间都被限流:纯指数退避会让它们在完全相同的时刻(5s、10s、20s…)一起醒来、齐刷刷同时重试——又把上游打垮,然后又一起被拒,永远同步撞车。加一点随机偏移就把它们错开(decorrelate),重试摊平到一段时间里,而不是挤在同一刻。
一个细节:种子用 time_ns() ^ (计数器 × 黄金比常数),即使系统时钟很粗、多个调用几乎同时发生,每个也能拿到不同种子,保证真的错开。

模式 B · 调用抛异常(恢复全景)

当 create() 直接抛异常,进入 except Exception as api_error。这里不是简单地「重试或放弃」,而是一套按错误类型精准施救的恢复系统——先分类,再挑对应的修复策略,修好就 continue 重试,修不了才升级到故障转移,最后才放弃。

第一步:错误分类

classified = classify_api_error(api_error, provider, model, approx_tokens, context_length, num_messages)
# 得到一个结构化判断:
#   reason           —— 具体原因(rate_limit / billing / timeout / context_overflow / thinking_signature ...)
#   retryable        —— 能不能重试
#   should_compress  —— 该不该压缩上下文
#   should_fallback  —— 该不该切备用
#   is_auth          —— 是不是认证问题

后面所有分支都基于这个 classified 结果决策,而不是靠零散的字符串匹配。

恢复策略全景(按类别)

拿到分类后,依次尝试下列策略。每一类都力求「就地修好、原地重试」,把放弃留到最后。

类别典型场景 → 做法

图片被拒:provider 说「只支持文本」→ 剥掉所有图片、标记本会话 vision 不可用、纯文本重试

凭证/认证恢复凭证池轮换:429 时 _recover_with_credential_pool 换一把 key 继续

防无限循环:「一次性」标志

很多恢复策略只该试一次(试完没用就别再原地打转)。这靠 TurnRetryState(_retry)里的一堆 xxx_attempted 布尔位守着,例如:

if classified.reason == FailoverReason.thinking_signature and not _retry.thinking_sig_retry_attempted:
    _retry.thinking_sig_retry_attempted = True    # 标记「这条路走过了」
    ...strip reasoning_details...
    continue

下次同样的错再来,这个标志已为真,就跳过该策略、往下走别的路——避免在同一种修复上无限重试。

放弃前的最后努力 + 可操作指引

retry_count >= max_retries 时并不直接死:先 _try_recover_primary_transport(重建主连接、清 TCP 死连接)、再 _try_activate_fallback(切备用)。真的都不行,才落库返回失败字典,并根据症状给出针对性建议——比如检测到「流反复断」提示改用 execute_code 写大文件;检测到「思考超时」提示调 stale_timeout_seconds 或降 reasoning effort。返回里还带 failure_reason,让上层(如 kanban worker)能区分「限流墙」和「真失败」。

一句话看懂模式 B:先分类 → 按类别就地修好重试(编码/图片/签名/schema…)→ 修不了就换凭证/换提供商 → 还不行就压缩或退避重试 → 全都无效才带着人话指引放弃。每条修复路径都有「一次性」标志防止死循环。这套东西就是 Hermes「在各种奇葩上游故障下也能扛住」的底气。

重试状态对象

这一堆重试不是用零散变量管的,而是集中在 TurnRetryState 和几个 agent._xxx_retries 计数(在准备阶段已归零)。两者分工不同:一个是「只准试一次」的布尔守卫,一个是「允许试 N 次」的整数计数。

① TurnRetryState:一次性恢复守卫

agent/turn_retry_state.py · 每圈(每个 api_call_count)new 一个新实例

它是个 dataclass,收拢了模式 B 里那些恢复分支的「这条路我走过没有」标记。历史包袱:这些原本是 ~16 个零散的 *_attempted / has_retried_* 裸局部变量,声明在循环前、再穿过 2400 行循环体传来传去;TurnRetryState 把它们收进一个有名字、可单测的对象,循环里就地改(_retry.codex_auth_retry_attempted = True)。里面装两类字段:

组字段(布尔,默认 False)守的恢复策略
凭证 / OAuth 刷新codex_auth_retry_attempted、anthropic_auth_retry_attempted、nous_auth_retry_attempted、nous_paid_entitlement_refresh_attempted、copilot_auth_retry_attempted各家 401 刷新令牌,每家每圈只刷一次
格式 / 载荷修复thinking_sig_retry_attempted、invalid_encrypted_content_retry_attempted、image_shrink_retry_attempted、multimodal_tool_content_retry_attempted、oauth_1m_beta_retry_attempted、llama_cpp_grammar_retry_attempted剥思考签名 / 剥加密回放 / 缩图 / 降级工具图 / 关 1M beta / 剥 schema 正则
传输 / 限流 & 认证转移primary_recovery_attempted、has_retried_429、auth_failover_attempted重建主连接 / 429 重试一次 / 认证失败切备用链
重启信号(循环尝试后读它,决定要不要重建请求再来)restart_with_compressed_messages、restart_with_length_continuation、restart_with_rebuilt_messages压缩后重来 / length 截断续写 / 内容过滤流中断→回滚→换 provider 重发
刻意的边界:循环机制变量 retry_count / max_retries / max_compression_attempts 故意留作裸局部变量——它们是 while 的机械控制,不是「恢复记账」,放进对象只会多一层间接、不增清晰度。

② agent._xxx_retries:按次数的整数计数

另一半是挂在 agent 上的整数计数器(在准备阶段归零),管的是「可以多试几次」的错——和上面「只准一次」的布尔守卫互补:

计数针对上限
_invalid_tool_retries模型调了不存在的工具硬编码 3
_invalid_json_retriestool_call 参数不是合法 JSON硬编码 3
_empty_content_retries模型返回空内容硬编码 3
length_continue_retries响应被 length 截断,续写硬编码 3
truncated_tool_call_retriestool_call 被截断硬编码 3
_codex_incomplete_retriesCodex 响应不完整硬编码 3
compression_attempts循环内因超限触发的压缩次数硬编码 3(max_compression_attempts)
哪些次数能配、哪些不能——分两层:
① 外层循环上限 max_retries = 可配置。它来自 agent._api_max_retries,启动时从 config.yaml 的 agent.api_max_retries 读(默认 3,下限钳到 1;非法值回落 3)。这是包住整次 API 调用的传输/限流重试次数: 
agent:
  api_max_retries: 5   # 默认 3;设 1 = 单次不重试
② 上表这些内容级子计数 = 硬编码 3,不可配。它们是「模型犯了某类具体错误(调错工具 / JSON 非法 / 空内容 / length 截断…),给它 3 次自我纠正机会」的内建策略,直接是字面量 3,不走配置。
一句话记住:布尔守卫(TurnRetryState._*_attempted)防「同一种修复原地打转」;整数计数(agent._*_retries)给「可以多试几次的错」一个次数上限。只有外层 agent.api_max_retries 可配置,内容级子计数全部硬编码 3。两者都在准备阶段重置,合起来构成一轮对话完整的重试记账。

本篇要点

  • 消息组装一次、重试多次;api_messages 是临时拷贝,清洗只改它。
  • 发请求前的清洗大多为缓存命中(前缀逐字节稳定)和跨模型兼容而做。
  • Anthropic 缓存断点可省约 75% 输入成本。
  • 请求默认走流式(健康检查防挂死)、可中断、外套执行中间件。
  • 失败分两种模式:A 响应无效(HTTP 200 空/畸形)与 B 调用抛异常(网络/4xx/5xx)。
  • 模式 B 先 classify_api_error 分类,再按类别就地修好重试(编码/图片/签名/schema)→ 换凭证/换提供商 → 压缩或退避 → 全无效才带指引放弃。
  • 恢复路径有两种防死循环记账:格式/认证类用「一次性」布尔标志(TurnRetryState),内容级错误用整数计数(硬编码 3);外层 max_retries 可配(agent.api_max_retries)。退避睡眠 0.2s 分片 + 可中断 + 每 30s 报活。
