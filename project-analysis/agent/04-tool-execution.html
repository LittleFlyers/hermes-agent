工具执行

模型返回 tool_calls 后,Hermes 怎么把这些「我要调工具」变成真实执行,再把结果喂回去?这一篇拆工具执行的完整流水线:并发还是串行、每个调用过哪些关卡、结果如何回灌。

分支:有工具就执行,没工具就收尾

response = client.chat.completions.create(...)   # 第 3 篇
if assistant_message.tool_calls:
    agent._execute_tool_calls_concurrent(assistant_message, messages, ...)
    # 或 _execute_tool_calls_sequential(...)
    continue                          # 回到循环顶部,带着工具结果再问模型
else:
    final_response = assistant_message.content   # 纯文本 → 第 5 篇收尾
agent/conversation_loop.py → agent/tool_executor.py

两个执行函数都是 run_agent.py 上的薄转发,真身在 agent/tool_executor.py:

  • execute_tool_calls_concurrent —— 线程池并发执行,结果按原始顺序收集再 append。
  • execute_tool_calls_sequential —— 逐个串行执行。
为什么结果要「按原始顺序」?因为 API 要求 tool 结果的顺序对应 assistant 发出的 tool_calls 顺序。并发执行打乱了完成时间,但收集时必须还原成原序,否则破坏消息结构(角色交替铁律)。

执行前:一次性中断预检

if agent._interrupt_requested:
    for tc in tool_calls:
        messages.append(make_tool_result_message(
            tc.function.name,
            "[Tool execution cancelled — 被用户中断]",
            tc.id))
        _flush_session_db_after_tool_progress(...)   # 落库
    return

如果用户在模型返回后、工具执行前按了中断:不执行任何工具,而是给每个 tool_call 都补一条「已取消」结果消息(保持结构合法),落库后直接返回。

每个工具调用过的关卡

对 tool_calls 里的每一个,执行前都要先解析参数、再依次过几道关卡。任意一关拦下,就用「拦截结果」代替真实执行:

  1. 解析参数 + 重置 nudge 计数

    取出 function_name 和参数。若调的是 memory 工具,把 _turns_since_memory 清零(刚复盘过,不必再提醒)。

  2. 越权检查(out-of-scope)

    若这个工具不在本会话授予的工具集里(子 agent、kanban worker、受限网关会话),在任何钩子/守卫之前就拒绝,并发一条 post_tool_call 终态钩子记录。

  3. pre_tool_call 插件钩子

    hermes_cli.plugins · pre_tool_call

    插件可在执行前拦截。被拦则取 get_pre_tool_call_block_message() 作为结果,不执行真实工具。

  4. 工具守卫(guardrail)

    agent._tool_guardrails.before_call(name, args)

    守卫策略判断是否放行。不放行则生成 guardrail_block_result,标记 blocked_by_guardrail=True,发终态钩子(error_type="guardrail_block")。

  5. 分发执行

    model_tools.handle_function_call(name, args, ...) → tools/registry.py

    过了前面所有关卡,才真正分发到工具注册表执行(终端、文件、网页…)。结果是 JSON 字符串。这一步的细节属于「工具系统」模块,本模块只需知道它是落点。

  6. 审批 / sudo 回调

    需要用户批准的工具(危险命令、写操作)在这里通过线程本地回调请求批准。

  7. post_tool_call 钩子

    执行后发出后置钩子,供插件观察/记录,以及网关推送工具完成事件。

  8. 按原序回灌结果

    messages.append(make_tool_result_message(name, result, tool_call_id))

    把结果包成 {"role":"tool", "tool_call_id":..., "content":...} append 进 messages,顺序对齐原始 tool_calls。

  9. 工具进度落库

    _flush_session_db_after_tool_progress(...)

    每执行完一个工具就刷一次 SQLite,保证长工具链中途崩溃也能恢复到最近进度。

并发执行如何保证顺序

execute_tool_calls_concurrent 的做法是:先把所有调用解析 + 过关卡(parsed_calls 列表,保留原始索引),再用线程池并发跑真实执行,最后按原索引把结果 append。所以:

阶段是否并发
解析参数 + 钩子/守卫前置检查串行(保证确定性)
真实工具执行并发(线程池)
结果回灌 messages按原始顺序串行 append

回灌后:带着结果再问一次

所有工具结果 append 进 messages 后,执行函数返回,主循环 continue 回到顶部。下一圈第 3 篇的清洗会把这些新增的 tool 消息一起发给模型——于是模型「看到」了工具结果,继续推理。这就是多轮工具调用能连续进行的根本。

flowchart LR
    A[模型返回 tool_calls] --> B[解析 + 过关卡
串行]
    B --> C[线程池并发执行]
    C --> D[按原序回灌 messages]
    D --> E[每个工具后落库]
    E --> F[continue 回循环顶]
    F --> G[下一圈带工具结果再问模型]

本篇要点

  • 有 tool_calls 就执行并 continue;纯文本则进收尾。
  • 真实执行可并发,但结果必须按原序回灌,守住角色交替铁律。
  • 每个调用执行前过四道关:越权 → pre 钩子 → 守卫 → 分发。
  • 真正的执行落点是 handle_function_call → registry(属工具系统模块)。
  • 每个工具执行后立即落库,长工具链也能崩溃恢复。
