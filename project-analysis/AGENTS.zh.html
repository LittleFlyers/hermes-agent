Hermes 是什么

Hermes 是一个个人 AI agent,同一套 agent 内核同时运行在 CLI、消息网关(Telegram、Discord、Slack 以及约 20 个其他平台)、TUI 和 Electron 桌面应用上。它能跨会话学习(记忆 + 技能)、把任务委派给子 agent、运行定时任务,并驱动真实的终端和浏览器。它的扩展主要依靠插件和技能,而非不断膨胀内核。

有两条根本属性贯穿了几乎每一个设计决策,也是审视任何改动时应有的视角:

每会话的 prompt 缓存神圣不可侵犯。 长期存在的会话每一轮都会复用缓存前缀。任何改动历史上下文、中途切换工具集、或重建 system prompt 的做法都会让缓存失效,使用户成本成倍上升。我们绝不这么做——唯一的例外是上下文压缩。
内核是「窄腰」,能力则生长在边缘。 我们每新增一个模型工具,它都会在每次 API 调用时被下发,因此新增一个核心工具的门槛极高。绝大多数新能力都应以 CLI 命令 + 技能、服务门控工具或插件的形式落地,而不是直接堆进核心。

贡献准则——准则的用法

这是项目的意图层。有两种用法:

  1. 对人类和你自己的工作而言——什么会被合并、什么会被拒绝,从而让贡献瞄准靶心。
  2. 对自动化审查(triage 清扫器)而言——指导何时可以按三个允许的理由(implemented_on_main、cannot_reproduce、incoherent)安全关闭 PR,以及同样重要的——何时不关闭。基于品味的"我们不想要/超范围"式关闭不是自动化决策,那留给人类维护者。清扫器在这里的职责是识别设计意图、避免错误地关闭一个合法贡献,而非自己做出"不实现"的判断。

把握分寸

Hermes 的交付量非常大——多数合入是针对真实反馈行为的 bug 修复,而产品的覆盖面(平台、渠道、提供商、模型、桌面/TUI 特性)始终在有意识地积极扩张。下面所说的克制只针对一处:核心 agent + 模型工具 schema——也就是每一次新增都要在每次 API 调用上持续付费的地方。"最小开销"约束的是一个能力以何种方式接入核心,而不是产品能否生长。我们在边缘大胆扩张,在窄腰处保持克制。

想要什么

  • 把真实 bug 修好。 大部分合入都是针对真实反馈症状的 fix(...)。一个好的修复能在当前 main 上复现症状,指出它出现的确切行,并修复整个 bug 类别——包括那些同类的调用路径——而不只是报告者刚好撞上的那一处。
  • 在边缘扩展覆盖面。 新的平台适配器、渠道、提供商、模型、桌面/TUI/dashboard 特性都受欢迎且经常落地,包括大型改动(新消息渠道、会话上限特性、Windows PTY 桥)。产品的广度本身就是目标,不是开销负担——前提是它接入既有的 setup/config 体验(hermes tools、hermes setup、自动安装),而不是甩出一个裸环境变量。
  • 把「上帝文件」重构成清爽的模块。 从 cli.py / run_agent.py / gateway/run.py 中抽出数千行的代码块,拆成职责单一的 mixin 或模块,是受欢迎的工作,哪怕 diff 巨大且机械。"每一行都能追溯到需求"这条标准适用于特性 PR;而一个声明为重构的 PR,其需求本身就是这次抽取。
  • 保持内核窄。 新模型工具是昂贵的例外——每个工具都在每次 API 调用时下发。优先顺序:扩展现有代码 → CLI 命令+技能 → 服务门控工具(check_fn)→ 插件 → MCP 目录中的服务器 → 新核心工具(最后手段)。
  • 扩展,而非重复。 在新增模块/管理器/钩子前,先检查既有基础设施是否已覆盖。当多个 PR 集成同一类别时,设计一个共享接口,而非逐个合并。
  • 行为契约高于快照。 测试应断言两份数据必须如何关联(不变量),而非冻结当前值(模型列表、配置版本字面量、枚举计数)。
  • 端到端验证,而非只有绿色单元 mock。 凡涉及解析链、配置传播、安全边界、远程后端、文件/网络 I/O 的,用真实路径、真实 import、对临时 HERMES_HOME 演练。Mock 会藏住集成 bug。
  • 缓存、交替、不变量安全。 保持 prompt 缓存、严格消息角色交替(绝不连续两条同角色;绝不在循环中途注入合成 user 消息),以及在会话生命周期内字节稳定的 system prompt。
  • 保留贡献者署名。 通过 cherry-pick(rebase-merge)救活外部工作,让作者身份在 git 历史中存续;能在别人基础上构建时,不要从零重写。

不想要什么(即便做得好也会拒绝)

  • 投机式的基础设施。 没有具体使用方的钩子、回调、扩展点。加一个钩子很容易,但等到插件开始依赖它,再想移除就难了。不过,只要贡献者有真实、明确的用例,即便使用方是另行交付的,这个钩子也不算投机。
  • 给非密钥配置新增 HERMES_* 环境变量。 .env 只放密钥(API key、token、密码)。所有行为设置——超时、阈值、特性开关、显示偏好——都进 config.yaml。机制需要时可在代码里桥接到内部环境变量,但面向用户的文档指向 config.yaml。拒绝那些让用户"在 .env 里设 X"的 PR,除非 X 是凭证。
  • 终端+文件已能做到、或本应做成技能时,却新增核心工具。 如果唯一障碍是远程后端上的文件可见性,修挂载,而非工具集。
  • 给指令型工具加「偏懒读取」的逃生口。 不要给那些加载了 agent 必须完整阅读内容的工具(技能、prompt、playbook)加 offset/limit 分页。模型会只读第一页就跳过其余。
  • 会摧毁它本该保护的特性的"修复"。 一个杀死了特性初衷的缓解,就是错误的缓解。在限制行为之前,先读原始提交的意图(git log -p -S)。
  • 无 opt-in 门控的外发遥测/用量归因。 在通用面向用户的 opt-in(配置门 + setup 提示 + hermes tools 开关)存在前,不加新分析、第三方标识标记或归因标签。打标签停放,不要合并。
  • 变更探测式测试、中途破缓存、无端到端证明却接线的死代码,以及碰核心文件的插件。 插件住在自己目录里、在我们提供的 ABC/钩子内工作;插件需要更多就拓宽通用插件表面,别在核心里特例化。
  • 把第三方产品/别人的项目集成进核心代码树。 可观测后端、厂商 SaaS 集成、分析面板等"别人家产品"的插件不放进本仓 plugins/ 下——它们会让我们为一个并不归我们所有的后端背上长期维护负担。这类插件应作为独立插件仓,由用户安装到 ~/.hermes/plugins/(或通过 pip 入口点),并在 Nous Research Discord(#plugins-skills-and-skins)推广。这是耦合与维护层面的取舍,而非质量评判——插件本身可以很优秀,却依然会被关闭。

在你称之为 bug 前——验证前提(以及何时不要关闭)

一个写得很好的 PR 被关闭的最常见原因并不是代码质量,而是它建立在错误前提之上,或是把一个有意的设计误当成了缺口。下面这些模式是双向的:既告诉人类审查者该重点审视什么,也告诉自动清扫器什么时候该把 PR 留给人类决定。

  • "有意设计,不是缺口。" 一个看似疏忽的限制往往是刻意的。在"修复"缺失链接或限制前,问问这种隔离是否就是设计。例:profiles 是有意独立的岛——一个给它加从默认 profile 实时配置继承的 PR 被关,因为把 profile 耦合在一起正是设计要防的(创建时复制的 --clone 路径已覆盖"从我的默认开始"的合法场景)。在假设某物未完成前,读原始提交意图(git log -p -S "<symbol>")。
  • "前提与 X 的真实工作方式不符。" PR 的理由常常建立在对既有机制的错误心智模型上。在接受这套理由前,先去追一遍真实代码与运行时行为。两个真实的关闭案例:限流"冷却期内重新探测"的 PR(熔断器只在已确认为空的账号配额桶上跳闸,所谓重探不过是反复敲打一个早已被证明为空的桶);还有一个用量累积的修复,它新加的分支在运行时根本不会执行,因为更早的守卫已经把它所依赖的状态弹出了。如果你既指不出 bug 出现的确切行,也证明不了修复改变了那一行的行为,那你就还没验证前提。
  • "这个修复本身是错的——那处缺失/省略是刻意为之。" 补上一个看似缺失的明显部件,可能会破坏这处省略原本在保护的东西。例如:恢复"缺失"的 __init__.py 后,一棵测试树就能作为点分包被导入,从而遮蔽了真正的插件,并在导入时把它的 register() 抹掉。那处缺失其实是不可或缺的。
  • "越界/复活了我们已越过的路线。" 超越已商定基线的范围蔓延,或复活维护者刻意关闭的方向,即便代码能跑也会被拒。把改动保持在实际商定的窄片,其余作为聚焦的后续。

主线:在写或合并修复前,对照代码库验证主张和意图。 当前 main 上确认的复现,加上修复在何处起作用的行级说明,胜过任何听起来合理的理由。对意图存疑时,问比出货更便宜。

开销阶梯(新增能力时如何决策)

越往下的档位,留下的永久开销越大。请选择能正确解决问题的最高一档(也就是开销最小的那一档):

档方式说明
1扩展现有代码新能力只是既有功能的一个变体,零新增开销。
2CLI 命令 + 技能当配置/状态/基础设施能用 shell 命令表达时,让 agent 在技能引导下执行 hermes <子命令>。对模型工具零开销。订阅、定时任务、服务搭建的首选方案。例:hermes webhook、hermes cron、hermes tools。
3服务门控工具(check_fn)需要结构化的参数/返回值,且仅在前置条件配置就绪时才出现,否则零开销。例:Home Assistant 工具(凭 token 门控)、记忆提供商工具。
4插件第三方/小众/用户特定能力,不进核心。住在 ~/.hermes/plugins/ 或 pip 包,运行时发现。
5MCP 服务器(进目录)能力确实需要做成工具、但又不属于核心基础时,优先做成 MCP 服务器并加进 MCP 目录。通过内置 MCP 客户端连接;对核心 schema 零永久开销,且任何 MCP host 都能复用。
6新核心工具仅当能力是基础的、对几乎所有用户广泛有用、且经终端+文件(或 MCP 服务器)不可达时。正确的核心工具例:terminal、read_file、web_search、browser_navigate。

当有 3 个以上的开放 PR 都想集成同一类别的东西(记忆后端、提供商、通知器)时,不要逐个合并——而应设计一个 ABC + 编排器,把既有的内置实现包装成第一个提供商,再把那些相互竞争的 PR 改造成针对该接口的插件。

TypeScript 风格

适用于 Hermes 的所有 TypeScript:桌面、TUI、网站及未来 TS 包。

  • 当状态被共享、复用或被远端 UI 读取时,优先用小 nanostore 而非组件 state。
  • 让每个特性拥有自己的 atom。聊天状态挨着聊天,shell 状态挨着 shell,共享状态在 src/store。
  • 从 atom 渲染的组件应用 useStore。非渲染动作应用 $atom.get() 读取。
  • 不要把状态穿过三层组件传递,当叶子可订阅 atom 时。
  • 持久化挨着拥有它的 atom。
  • 路由根保持薄。它们组合路由与 shell,不应变成控制器。
  • 不要写巨型 hook。一个 hook 应只管一件窄事。
  • 优先共置的动作模块而非隐藏的上帝 hook。
  • 纯副作用回调用简洁 void 形式:onState={st => void setGatewayState(st)}。
  • 异步 UI 处理器让意图显式:onClick={() => void save()}。
  • 公共 props 与共享对象形状优先 interface。避免对对象 props 用 type X = {…}。
  • props 扩展 React 原语:React.ComponentProps<'button'>、Omit<…>、Pick<…>。
  • 映射 id/路由/视图时,表驱动胜过条件梯。
  • src/app 拥有路由、页面、页面专属组件;src/store 拥有共享 atom;src/lib 拥有共享纯助手。

AIAgent 类(run_agent.py)

真实的 AIAgent.__init__ 接收约 60 个参数(凭证、路由、回调、会话上下文、预算、凭证池等)。下面是你通常会碰的最小子集——完整列表读 run_agent.py。

class AIAgent:
    def __init__(self,
        base_url=None, api_key=None, provider=None,
        api_mode=None,              # "chat_completions" | "codex_responses" | ...
        model="",                   # 空 → 稍后从 config/provider 解析
        max_iterations=90,          # 工具调用迭代(与子 agent 共享)
        enabled_toolsets=None, disabled_toolsets=None,
        quiet_mode=False, save_trajectories=False,
        platform=None,              # "cli"、"telegram" 等
        session_id=None,
        skip_context_files=False, skip_memory=False,
        credential_pool=None,
        # … 外加回调、thread/user/chat ID、iteration_budget、fallback_model、
        # checkpoints 配置、prefill_messages、service_tier、reasoning_config 等
    ): ...

    def chat(self, message: str) -> str:
        """简单接口——返回最终响应字符串。"""

    def run_conversation(self, user_message, system_message=None,
                         conversation_history=None, task_id=None) -> dict:
        """完整接口——返回含 final_response + messages 的 dict。"""

Agent 循环

核心循环在 run_conversation() 内——完全同步,带中断检查、预算追踪、一轮宽限调用:

while (api_call_count < self.max_iterations and self.iteration_budget.remaining > 0) \
        or self._budget_grace_call:
    if self._interrupt_requested: break
    response = client.chat.completions.create(model=model, messages=messages, tools=tool_schemas)
    if response.tool_calls:
        for tool_call in response.tool_calls:
            result = handle_function_call(tool_call.name, tool_call.args, task_id)
            messages.append(tool_result_message(result))
        api_call_count += 1
    else:
        return response.content

消息遵循 OpenAI 格式:{"role": "system/user/assistant/tool", …}。推理内容存于 assistant_msg["reasoning"]。

CLI 架构(cli.py)

  • Rich 做横幅/面板,prompt_toolkit 做带自动补全的输入。
  • KawaiiSpinner(agent/display.py)——API 调用时的动画表情,┊ 活动流显示工具结果。
  • load_cli_config() 合并硬编码默认 + 用户配置 YAML。
  • 皮肤引擎(hermes_cli/skin_engine.py)——数据驱动 CLI 主题;启动时从 display.skin 配置键初始化。
  • process_command() 是 HermesCLI 的方法——经 resolve_command() 在中央注册表解析后,按规范命令名派发。
  • 技能斜杠命令:agent/skill_commands.py 扫描 ~/.hermes/skills/,作为用户消息(非 system prompt)注入以保护 prompt 缓存。

斜杠命令注册表(hermes_cli/commands.py)

所有斜杠命令定义在中央 COMMAND_REGISTRY(CommandDef 对象列表)。每个下游消费者自动派生:

  • CLI——process_command() 经 resolve_command() 解析别名,按规范名派发。
  • 网关——GATEWAY_KNOWN_COMMANDS frozenset 用于钩子发射,resolve_command() 用于派发。
  • 网关帮助——gateway_help_lines() 生成 /help 输出。
  • Telegram——telegram_bot_commands() 生成 BotCommand 菜单。

新增工具

新增任何工具前,先把开销问题想清楚(见「开销阶梯」):绝大多数能力不该做成核心工具。自定义或仅本地使用的工具不要去改 Hermes 核心,请走插件路线:建 ~/.hermes/plugins/<name>/plugin.yaml 和 __init__.py,用 ctx.register_tool(...) 注册。

只有当用户明确要贡献一个该随基础系统一起发布的新核心工具时,才用下面的内置路线。核心工具需改 2 个文件:

1. 建 tools/your_tool.py

import json, os
from tools.registry import registry

def check_requirements() -> bool:
    return bool(os.getenv("EXAMPLE_API_KEY"))

def example_tool(param, task_id=None) -> str:
    return json.dumps({"success": True, "data": "..."})

registry.register(
    name="example_tool",
    toolset="example",
    schema={"name": "example_tool", "description": "...", "parameters": {...}},
    handler=lambda args, **kw: example_tool(param=args.get("param",""), task_id=kw.get("task_id")),
    check_fn=check_requirements,
    requires_env=["EXAMPLE_API_KEY"],
)

2. 加进 toolsets.py

加进 _HERMES_CORE_TOOLS(所有平台)或新工具集。此步必需:自动发现会导入工具并注册 schema,但工具只有当其名出现在某工具集时才暴露给 agent。_HERMES_CORE_TOOLS 不是死代码——它是每个平台基础工具集继承的默认包。

自动发现:任何含顶层 registry.register() 调用的 tools/*.py 自动导入。所有 handler 必须返回 JSON 字符串。schema 描述提及路径时用 display_hermes_home() 使其 profile 感知;持久状态用 get_hermes_home()。agent 级工具(todo、memory)在 handle_function_call() 前被 run_agent.py 拦截。

插件

Hermes 有两个插件表面,都住在仓库 plugins/ 下,与用户安装的 ~/.hermes/plugins/ 及 pip 入口点一同被发现。

通用插件(hermes_cli/plugins.py + plugins/<name>/)

PluginManager 从 ~/.hermes/plugins/、./.hermes/plugins/ 及 pip 入口点发现。每个插件暴露 register(ctx),可:注册 Python 回调生命周期钩子(pre_tool_call、post_tool_call、pre/post_llm_call、on_session_start/end);经 ctx.register_tool(...) 注册新工具;经 ctx.register_cli_command(...) 注册 CLI 子命令。

发现时机坑:discover_plugins() 仅作为导入 model_tools.py 的副作用运行。不导入 model_tools 就读插件状态的路径必须显式调用它(幂等)。

记忆提供商插件(plugins/memory/<name>/)

可插拔记忆后端的独立发现系统。当前内置:honcho、mem0、supermemory、byterover、hindsight、holographic、openviking、retaindb。各实现 MemoryProvider ABC,由 agent/memory_manager.py 编排。生命周期钩子:sync_turn、prefetch、shutdown、可选 post_setup。

规则(Teknium,2026-05):插件禁止修改核心文件(run_agent.py、cli.py、gateway/run.py、main.py 等)。插件需要框架未暴露的能力,就拓宽通用插件表面(新钩子、新 ctx 方法),绝不在核心硬编码插件专属逻辑。无内树新记忆提供商(2026-05)、无内树新第三方产品插件(2026-06)——作为独立插件仓发布。

模型提供商插件(plugins/model-providers/<name>/)

每个推理后端(openrouter、anthropic、gmi、deepseek、nvidia…)都作为插件发布。各 __init__.py 在加载时调用 register_provider(ProviderProfile(...))。_discover_providers() 是一个懒加载、独立的发现系统,首次调用 get_provider_profile() 或 list_providers() 时才扫描。扫描顺序:仓库内置 → 用户 → 旧版 providers/<name>.py。同名的用户插件会覆盖内置的(后注册者胜出)。

技能

两个并行表面:skills/(默认可加载内置技能,按类别目录组织)与 optional-skills/(较重/小众,经 hermes skills install official/<category>/<skill> 显式安装)。

SKILL.md frontmatter

标准字段:name、description、version、author、license、platforms(OS 门控)、metadata.hermes.tags/category/related_skills/config。

技能创作标准(硬线)

  1. description ≤ 60 字符,一句话,以句号结尾。 陈述能力而非实现,无营销词,不重复技能名。
  2. SKILL.md 正文引用的工具须是原生 Hermes 工具或 MCP 服务器。 用反引号点名(terminal、read_file、patch、search_files 等)。不要点 agent 已经包装好的 shell 工具(grep→search_files,cat→read_file,sed→patch,find→search_files)。
  3. platforms: 门控须对照脚本实际 import 审计。 用 POSIX 专属原语(fcntl、termios、os.setsid、/proc 等)须声明平台。优先跨平台修复。
  4. author 先署人类贡献者。 外部贡献署真名 + GitHub handle 在前,"Hermes Agent"次之。
  5. SKILL.md 正文用现代章节序:标题、2-3 句意图、When to Use、Prerequisites、How to Run、Quick Reference、Procedure、Pitfalls、Verification。复杂技能 ~200 行,简单 ~100 行。
  6. 脚本进 scripts/,引用进 references/,模板进 templates/。 别让模型每次内联写解析器。
  7. 测试在 tests/skills/test_<skill>_skill.py,仅用 stdlib + pytest + mock,无网络。
  8. .env.example 新增隔离在清晰分隔块。 别碰周围文件。

全部工具集定义在 toolsets.py 的单个 TOOLSETS dict。每平台适配器选一个基础工具集(如 Telegram 用 "messaging");_HERMES_CORE_TOOLS 是多数平台继承的默认包。

当前键:browser、clarify、code_execution、cronjob、debugging、delegation、discord、discord_admin、feishu_doc、feishu_drive、file、homeassistant、image_gen、kanban、memory、messaging、moa、rl、safe、search、session_search、skills、spotify、terminal、todo、tts、video、vision、web、yuanbao。

经 hermes tools(curses UI)或 config 的 tools.<platform>.enabled/disabled 列表逐平台开关。

委派(delegate_task)

tools/delegate_tool.py spawn 一个带隔离上下文 + 终端会话的子 agent。默认父等子摘要再继续。background=true 时立即返回委派 id,结果稍后经异步完成队列重入会话。

两种形态:单个任务传 goal(可选附带 context、toolsets);批量(并行)则传 tasks:[...],每个任务都获得独立的并发子 agent,并发数由 delegation.max_concurrent_children(默认 3)限制。

角色:role="leaf"(默认)是职责聚焦的工作者,不能调用 delegate_task/clarify/memory/send_message/execute_code;role="orchestrator" 保留 delegate_task,可自己再 spawn 子 agent,由 orchestrator_enabled(默认 true)门控、max_spawn_depth(默认 2)限定深度。

耐久规则:后台 delegate_task 虽然脱离了当前轮,但仍然只在本进程内。需要在进程重启后仍能存活的工作,请用 cronjob 或 terminal(background=True, notify_on_complete=True)。

Curator(技能生命周期)

后台技能维护系统,追踪 agent 自建技能用量并自动归档陈旧者。用户永不丢技能;归档进 ~/.hermes/skills/.archive/,可恢复。

  • 核心:agent/curator.py(审查循环、自动转换、LLM 审查 prompt)+ agent/curator_backup.py(运行前 tar.gz 快照)。
  • CLI:hermes curator <verb>——status、run、pause、resume、pin、unpin、archive、restore、prune、backup、rollback。
  • 遥测:tools/skill_usage.py 拥有 .usage.json——use_count、view_count、patch_count、last_activity_at、state、pinned。

不变量:仅碰 created_by: "agent" 的技能;从不删除,最大破坏动作是归档;pinned 技能免于所有自动转换与 LLM 审查;skill_manage(action="delete") 拒绝 pinned。

Cron(定时任务)

cron/jobs.py(任务存储)+ cron/scheduler.py(tick 循环)。agent 经 cronjob 工具排程;用户经 hermes cron <verb> 或 /cron 斜杠命令。

排程格式:时长("30m"、"2h"、"1d")、"every" 短语("every 2h"、"every monday 9am")、5 字段 cron 表达式("0 9 * * *")、ISO 时间戳(一次性)。

每任务字段:skills、model/provider 覆盖、script(预运行数据采集脚本,stdout 注入 prompt)、context_from(链接任务输出)、workdir、多平台投递。

硬化不变量:cron 会话 3 分钟硬中断;catchup 窗口为任务周期一半(120s–2h);grace 窗口 120s;文件锁 ~/.hermes/cron/.tick.lock 防重复 tick;默认 skip_memory=True。cron 投递不镜像进目标网关会话,落在自己的 cron 会话以保持消息角色交替。

Kanban(多 agent 工作队列)

SQLite 支撑的耐久看板,让多个 profile/worker 协作共享任务。用户经 hermes kanban <verb> 驱动;worker 经专用 kanban_* 工具集驱动(不在任务内时零 schema 脚印)。

  • CLI:hermes kanban 含 init、create、list、show、assign、link、comment、complete、block、archive、tail 等。
  • Worker 工具集:kanban_show、kanban_complete、kanban_block、kanban_heartbeat、kanban_comment、kanban_create、kanban_link。
  • Dispatcher:长寿循环(默认每 60s),回收陈旧 claim、提升就绪任务、原子 claim、spawn 指派 profile。默认网关内跑(kanban.dispatch_in_gateway: true)。

隔离模型:board 是硬边界(worker 用 HERMES_KANBAN_BOARD 钉死);tenant 是 board 内软命名空间。同任务连续 failure_limit(默认 2)次非成功后,dispatcher 自动 block 防自旋。

重要策略

Prompt 缓存不可破。 不要改动历史上下文、不要中途换工具集、不要中途重载记忆或重建 system prompt。破缓存大幅抬高成本。唯一改动上下文的时机是上下文压缩。改 system-prompt 状态的斜杠命令须缓存感知:默认延迟失效(下会话生效),opt-in --now 立即失效。

后台进程通知(网关)

terminal(background=true, notify_on_complete=true) 时,网关跑 watcher 检测完成并触发新轮。经 display.background_process_notifications 控制详细度:all(运行输出+最终)、result(仅最终)、error(仅退出码≠0)、off(无)。

Profiles:多实例支持

Hermes 支持 profiles——多个完全隔离的实例,各有自己的 HERMES_HOME。核心机制:main.py 的 _apply_profile_override() 在任何模块导入前设 HERMES_HOME。

  1. 所有 HERMES_HOME 路径用 get_hermes_home()。 绝不硬编码 ~/.hermes。
  2. 面向用户消息用 display_hermes_home()。
  3. 模块级常量可以——它们在导入时缓存,那已在 profile 覆盖之后。
  4. mock Path.home() 的测试须同设 HERMES_HOME。
  5. 网关平台适配器应用 token 锁——以唯一凭证连接时,在 connect/start 调 acquire_scoped_lock(),在 disconnect/stop 调 release_scoped_lock()。
  6. Profile 操作锚 HOME 而非 HERMES_HOME——_get_profiles_root() 返回 Path.home()/".hermes"/"profiles",让 hermes -p coder profile list 无论哪个 profile 活动都能看到全部。

已知坑

  • 别硬编码 ~/.hermes 路径。 代码路径用 get_hermes_home(),用户消息用 display_hermes_home()。硬编码破坏 profiles(PR #3575 修了 5 个此类 bug)。
  • 别引入新 simple_term_menu 用法。 用 curses(stdlib),因前者在 tmux/iTerm2 有幽灵重影 bug。见 hermes_cli/tools_config.py。
  • spinner/显示代码别用 \033[K(ANSI 擦到行尾)。 在 prompt_toolkit 的 patch_stdout 下会漏成字面 ?[K。用空格填充 f"\r{line}{' ' * pad}"。
  • _last_resolved_tool_names 是 model_tools.py 的进程全局。 _run_single_child() 在子 agent 执行前后保存/恢复它,子运行期间可能暂时陈旧。
  • schema 描述里别硬编码跨工具引用。 别让一个工具的 schema 提别工具集的工具名(可能不可用导致模型幻觉调用)。需要时在 get_tool_definitions() 动态加。
  • 网关有两道消息守卫,审批/控制命令须绕过两者。 base 适配器把消息排进 _pending_messages;网关 runner 拦截 /stop、/new、/queue、/status、/approve、/deny。须到达 runner 的新命令须绕过两道并内联派发。
  • 陈旧分支的 squash 合并会悄悄回滚近期修复。 squash 前确保分支与 main 同步,合并后用 git diff HEAD~1..HEAD 验证,意外删除是红旗。
  • 别在无端到端验证下接线死代码。 把未用模块接进活路径前,用真实 import 对临时 HERMES_HOME 端到端测真实解析链。
  • 测试不得写 ~/.hermes/。 conftest.py 的 _isolate_hermes_home autouse fixture 重定向到临时目录。profile 测试还须 mock Path.home()。

测试

始终用 scripts/run_tests.sh——别直接调 pytest。脚本强制与 CI 一致的环境(清空凭证变量、TZ=UTC、LANG=C.UTF-8、-n auto xdist、子进程隔离插件)。在有 API key 的多核机上直接 pytest 会与 CI 漂移。
scripts/run_tests.sh                                  # 全量,CI 对齐
scripts/run_tests.sh tests/gateway/                   # 单目录
scripts/run_tests.sh tests/agent/test_foo.py::test_x  # 单测
scripts/run_tests.sh -v --tb=long                     # 透传 pytest 标志
scripts/run_tests.sh --no-isolate tests/foo/          # 关子进程隔离(更快,调试用)

每测子进程隔离

每个测试经 tests/_isolate_plugin.py 在新 spawn 的 Python 子进程跑。模块级 dict/set 与 ContextVar 不会从一个测试泄漏到下一个。用 multiprocessing.get_context("spawn")(Linux/macOS/Windows 通用)。每测开销 ~0.5–1.0s,xdist 摊销。isolate_timeout 每测 30s 上限。--no-isolate 关闭。

别写变更探测式测试

一个测试若在预期会变的数据(模型目录、配置版本号、枚举计数、硬编码模型列表)更新时失败,它就是变更探测器,只会让例行更新破坏 CI。

# 别写:目录快照——每次模型发布就坏
assert "gemini-2.5-pro" in _PROVIDER_MODELS["gemini"]
assert DEFAULT_CONFIG["_config_version"] == 21

# 要写:行为/不变量
assert "gemini" in _PROVIDER_MODELS
assert len(_PROVIDER_MODELS["gemini"]) >= 1
assert raw["_config_version"] == DEFAULT_CONFIG["_config_version"]
for m in _PROVIDER_MODELS["huggingface"]:
    assert m.lower() in DEFAULT_CONTEXT_LENGTHS_LOWER

规则:测试若读起来像当前数据快照,删掉;若读起来像两份数据如何关联的契约,保留。推改动前跑全量。
